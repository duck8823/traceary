# Spec 1195: Antigravity Capability Detection

## Requirement summary

Add `traceary doctor --client antigravity --json` to detect whether Antigravity (Google's successor to Gemini CLI) is installed and whether a supported public headless/hook/package surface exists for Traceary integration.

As of v0.21.0, the local environment has `/Applications/Antigravity.app` (version 2.1.4, bundle ID `com.google.antigravity`) but no `antigravity` command on PATH and no confirmed public hook/CLI contract. The expected detection outcome is `tool_unavailable`, not `auth_prompt`.

**Risk level: Medium** — no new persistence, no schema change, no hook wiring. Risk is limited to incorrect capability state reporting and test coverage gaps.

## Conceptual model

Antigravity capability has four states:

| State | Meaning |
|---|---|
| `not_installed` | No app bundle and no CLI on PATH |
| `tool_unavailable` | App/CLI found but no supported public headless/hook/package surface confirmed |
| `not_authenticated` | Supported surface confirmed but credentials/config not set up (future) |
| `available` | Supported surface confirmed and authenticated/configured (future) |

The detection is split into two layers:

1. **Probe collection** (`antigravityCapabilityProbe`): gathers raw observed facts — `CLIFound`, `BundleFound`, `SupportedSurfaceConfirmed`, `AuthenticatedOrConfigured`. I/O lives here (PATH lookup, `os.Stat`).
2. **State mapping** (`antigravityProbeToState`): pure function over the probe struct. No I/O. Fully testable for all four states including future `not_authenticated` and `available`.

The production `detectAntigravityCapability` wires injected predicates into a probe and delegates to `antigravityProbeToState`. As of v0.21.0, `SupportedSurfaceConfirmed` is always false, so the reachable states in production are `not_installed` and `tool_unavailable`.

## Responsibility assignment

| Concern | Owner |
|---|---|
| Observed-fact collection | `antigravityCapabilityProbe` struct + `detectAntigravityCapability` (I/O boundary) |
| Pure state mapping | `antigravityProbeToState` (no I/O; testable for all four states) |
| Production wiring | `inspectAntigravityCapability` |
| Doctor check construction | `buildAntigravityCapabilityCheck` |
| `--client antigravity` routing bypass | `resolveDoctorClients` in `doctor.go` |
| Client loop skip for Antigravity | `buildDoctorReport` in `doctor.go` |

## Boundaries / interfaces

- **Doctor only**: `--client antigravity` is accepted by `doctor` only. `hooks install/print/guide` with `--client antigravity` remains unsupported.
- **No hook orchestrator calls**: `hooksOrchestrator.NormalizeClient` and `ResolveInstallPath` are NOT called for `antigravity`. The doctor loop uses a continue-skip to bypass hook config checks.
- **No app launch, no credential read**: `detectAntigravityCapability` uses only `exec.LookPath` and `os.Stat` (via injected functions) to populate `antigravityCapabilityProbe.CLIFound` and `BundleFound`. It reads no credentials, tokens, keychains, or private app storage. `AuthenticatedOrConfigured` stays `false` in v0.21.0 production wiring.
- **Probe-to-state separation**: `antigravityProbeToState(p antigravityCapabilityProbe)` is a pure function. Tests can set any combination of probe fields to exercise `not_authenticated` and `available` paths without mocking filesystem or PATH. Production sets `SupportedSurfaceConfirmed=false`, making `not_authenticated` and `available` unreachable until a future contract is confirmed.
- **Bundle paths are OS-specific**: Only macOS paths (`/Applications/Antigravity.app`) are included in v0.21.0.
- **Check name**: `antigravity-capability`. Falls into the `Hooks` section in the doctor report (no hook config installed; the section name reflects the category is about hooks/integration state).

## Behavior tests

1. **not_installed → WARN without `tool_unavailable`**: `detectAntigravityCapability` with no app bundle and no CLI returns `not_installed`; `buildAntigravityCapabilityCheck` produces `status=warn` with a message that does NOT contain `tool_unavailable`.
2. **installed-but-no-surface → WARN with `tool_unavailable`**: `detectAntigravityCapability` with app bundle found but no CLI returns `tool_unavailable`; `buildAntigravityCapabilityCheck` produces `status=warn` with message containing `tool_unavailable`.
3. **End-to-end JSON** — `doctor --client antigravity --json` reports `clients:["antigravity"]`, includes exactly one check named `antigravity-capability`, no `antigravity-config` check, no Claude/Codex/Gemini-only checks.
4. **Default clients unchanged** — `doctor --json` (no `--client`) still reports `clients:["claude","codex","gemini"]`.

## Risks / rollback

- **Check section classification**: `antigravity-capability` goes to "Hooks" section by default in `doctorSectionNameForCheck`. This is slightly imprecise (no hook is configured yet) but consistent with how unsupported-client states appear in other checks.
- **macOS-only bundle path**: On Linux/Windows with no `antigravity` on PATH, the state is `not_installed`. This is correct behavior — Antigravity has no confirmed Linux/Windows install path.
- **Future state promotion**: When Google exposes a supported CLI or hook contract, the detector adds a PATH/config check returning `available`. No schema or persistence changes are required.
- **Rollback**: Delete `doctor_antigravity.go`, revert the two hunks in `doctor.go`. No persistence, no migration, no hook config written.
