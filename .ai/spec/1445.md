# Issue #1445: v0.31 copied-store dogfood and retention status

## Requirements and risk

- High-risk release-safety work: no QA may touch the only copy of a live store.
- `doctor` / `status` must report archive/backup capacity only when the operator explicitly supplies roots; inspection is read-only and never creates a plan file or applies retention.
- Default install/update remains non-destructive. Raw-body pruning and archive/backup capacity apply stay hidden, manual, and exact-plan opt-in for v0.31.
- Dogfood evidence must cover wrong confirmation, interruption, retry, restore, rollback, logical/allocated size, query behavior, and retained recovery-point integrity.
- Release-blocking QA findings become child follow-up issues and are fixed before release.

## Concept model

| Concept | State | Rule |
|---|---|---|
| CapacitySnapshot | class, root, file counts, logical/allocated bytes, verification counts, recovery floor | derived from one read-only inventory snapshot |
| CapacityReadiness | `empty`, `ready`, `indeterminate` | non-empty inventory is ready only with a verified current-generation floor and no blockers |
| DogfoodRun | copied/synthetic paths, before/after evidence, recovery result | never points at the sole live store |
| PolicyDecision | automatic/manual status per payload class | v0.31 keeps raw-body and file capacity mutation manual and opt-in |

## Responsibilities and boundaries

| Responsibility | Owner | Boundary |
|---|---|---|
| inventory and inode/content verification | filesystem adapter | existing `FileRetentionInventory` port |
| aggregate capacity/readiness without policy mutation | file-retention application use case | `InspectCapacity` consumer interface |
| CLI flags, localized diagnosis, warning severity | doctor presentation | `--archive-root`, `--backup-root` |
| copied-store drills and policy decision | bilingual operations document | version-controlled evidence summary plus PR QA log |

The doctor must not reconstruct plan policy, parse private filesystem state, or call apply. The application use case owns aggregation; the adapter remains responsible only for verified I/O evidence.

## Observable behavior tests

1. Capacity inspection sums logical bytes and known allocated bytes from exactly one class snapshot.
2. Empty inventory reports `empty` without warning.
3. A verified current-generation floor with no blockers reports `ready` and names the floor.
4. Missing floor or blocker reports `indeterminate`; unverified counts remain visible.
5. Doctor without capacity-root flags preserves existing output and performs no extra inventory.
6. Doctor with a root emits a Database-section check containing counts, byte totals, readiness, and the explicit manual-plan hint.
7. Inventory errors become a warning/failure check and never trigger mutation.

## Rollback and release gate

- Code rollback removes only read-only status reporting; existing hidden plan/apply contracts remain unchanged.
- Dogfood uses disposable copies and synthetic roots. Rollback means deleting the disposable target or restoring the retained verified backup into a new disposable DB, never replacing the live DB during QA.
- Any wrong-path deletion, metadata/query regression, failed retry convergence, or failed retained-floor restore blocks v0.31 and requires a follow-up issue/PR before release.

## Structure self-review

- Core readiness rules are not placed in Cobra handlers.
- The status DTO carries behavior-relevant aggregates, not an infrastructure DTO leak.
- No boolean mutation flag is added; inspection and apply remain separate interfaces.
- Tests assert public status/doctor output and recovery behavior, not private call order.
