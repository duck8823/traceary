# Spec 1196: Antigravity unavailable-package decision (v0.21.0)

Parent: #1194 · Predecessor: #1195 (capability detection)

## Requirement summary

#1196 asked to "add an Antigravity hook and dogfood integration package." The
acceptance criteria are conditional:

- Add an Antigravity package **only if** #1195 confirms a supported public
  hook/plugin/MCP/headless CLI surface.
- If #1195 confirms **no** supported surface, do **not** ship a fake package.
  Instead, document the intentionally omitted package and ensure
  doctor/release guidance reports Antigravity unsupported/unavailable for
  v0.21.0.
- Tests cover integration metadata generation **or** the explicit
  unavailable-package / doctor path.

#1195 outcome (confirmed): local discovery found `/Applications/Antigravity.app`
(version 2.1.4, bundle ID `com.google.antigravity`), but **no** supported public
headless / hook / plugin / MCP / package surface is confirmed. There is no
`antigravity` binary on PATH and no install mechanism equivalent to
`gemini extensions install`.

**Decision:** v0.21.0 intentionally ships **no** Antigravity package, hook
config, generated package metadata, or release asset. Shipping a package now
would require fabricating a hook contract that Google has not published. The
explicit unavailable-package / doctor path is implemented instead.

**Risk level: Medium** — no new persistence, no schema change, no hook wiring.
The change is a doctor message correction plus documentation and release-guidance
updates. Risk is limited to (a) doctor message drift vs. docs and (b) docs
implying package work is still pending.

## Conceptual model

The decision rides on the four-state capability model already established in
#1195 (`doctor_antigravity.go`):

| State | Meaning |
|---|---|
| `not_installed` | No app bundle and no CLI on PATH |
| `tool_unavailable` | App/CLI found but no supported public headless/hook/package surface confirmed |
| `not_authenticated` | Supported surface confirmed but not authenticated/configured (future) |
| `available` | Supported surface confirmed and configured (future) |

In v0.21.0 production wiring, `SupportedSurfaceConfirmed` is always `false`, so
the only reachable states are `not_installed` and `tool_unavailable`. #1196 does
**not** add states, persistence, or wiring. It changes the **meaning conveyed by
the `tool_unavailable` message**: from "hook/package implementation is tracked in
#1196" (pending work) to "v0.21.0 intentionally ships no Antigravity
hook/package/release asset; future support requires a supported public CLI/hook
contract" (a settled decision, not a backlog item).

## Boundaries / interfaces

- **No package files added.** No `integrations/antigravity-*`, no
  `.codex-plugin`/`gemini-extension`-equivalent manifest, no generated metadata,
  no release asset, no fake hook config JSON. The Antigravity surface in the repo
  remains: doctor capability detection (code) + documentation only.
- **Doctor only.** `--client antigravity` stays accepted by `doctor` only;
  `hooks install/print/guide --client antigravity` remains unsupported.
- **Message contract.** The `tool_unavailable` message must (1) keep the
  `tool_unavailable` token (consumed by tooling/tests and JSON readers), (2)
  state that v0.21.0 intentionally ships no Antigravity hook/package/release
  asset, and (3) state that future support requires a supported public CLI/hook
  contract. It must **not** point at #1196 as outstanding package work.
- **State semantics preserved.** `not_installed`, `not_authenticated`, and
  `available` messages are unchanged. Only the `tool_unavailable` message body
  changes.
- **Release asset boundary.** No GoReleaser change; the release pipeline already
  publishes no Antigravity asset. Docs are updated to make the omission explicit
  and intentional rather than "pending."

## Behavior tests

1. **tool_unavailable states the decision** — `buildAntigravityCapabilityCheck(
   antigravityStateToolUnavailable)` returns `status=warn`, message contains
   `tool_unavailable`, contains "intentionally ships no Antigravity", contains
   "supported public CLI/hook contract", and does **not** contain `#1196`.
2. **State semantics preserved** — `not_installed` still WARNs without
   `tool_unavailable`; `not_authenticated` still WARNs with `not_authenticated`
   and without a "reads credentials" claim; `available` still PASSes. (Existing
   #1195 tests retained.)
3. **Probe → state mapping unchanged** — all `antigravityProbeToState` and
   `detectAntigravityCapability` table cases from #1195 still pass.

## Risks / rollback

- **Doc/message drift.** The decision narrative now lives in both the doctor
  message and several docs. Mitigation: tests pin the message contract; docs
  reference the same `tool_unavailable` state and decision.
- **Future re-enable.** When Google publishes a supported public CLI/hook
  contract, the path is: set `SupportedSurfaceConfirmed=true` in production
  wiring, add the `available`/`not_authenticated` checks (already modeled), then
  add the package under a new issue. No schema/persistence migration is needed,
  and this spec's decision is explicitly scoped to "until a supported public
  CLI/hook contract exists."
- **Rollback.** Revert the `tool_unavailable` message hunk and the test hunk;
  docs revert independently. No persistence, migration, or hook config is
  written by this change.

## Release / review policy note (v0.21.0)

This change ships under the current v0.21.0 review policy:

- Gemini review is **not** required — Gemini CLI is retired/unavailable.
- Codex implementation/review is **not** required when local policy / user
  instruction disables it.
- The gate is **Claude review + local dogfood + tests/CI**.
- Antigravity doctor state remains `tool_unavailable` unless a future supported
  public CLI/hook contract appears.
