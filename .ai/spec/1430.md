## Structure-Behavior Design Note

### Requirement summary
- Purpose: give persisted sessions an explicit runtime mode and one typed effective terminal reason so one-shot finalization can be implemented without treating interactive sessions as disposable.
- Current state: a session has only nullable `ended_at`; mode is implicit and terminal outcomes collapse into a generic `session_ended` boundary.
- Expected behavior: new sessions have an explicit non-zero mode, terminal transitions are applied once, same-reason redelivery is an observable no-op, different reasons fail closed, and legacy rows remain readable without inventing an outcome.
- Out of scope: host-hook finalization, transcript inference, idle closure, stale-session mutation, and parent aggregate reporting. Those remain in #1436 and #1434.

### Conceptual model
| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Runtime mode | interactive, one_shot, resumed, background | Determines which lifecycle contract owns closure | Empty/unknown values never mean interactive or one-shot |
| Terminal reason | success, failure, timeout, signal, aborted_stream, legacy_unknown | Explains the effective end state | Active sessions have no reason; ended sessions have exactly one |
| Terminal transition | applied, already_applied, conflict | Applies or compares a proposed reason | Same reason never overwrites the first timestamp/summary; different reason never mutates state |
| Legacy session | explicit interactive mode; ended rows use legacy_unknown | Restores pre-v0.30 data conservatively | Never inferred as one-shot and never assigned a fabricated success/failure |

### Responsibility assignment
| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| Validate runtime/reason vocabulary | domain value objects | Lifecycle vocabulary changes | SQLite/CLI must not duplicate enum rules |
| Enforce one effective terminal state | Session aggregate | Session lifecycle invariant changes | Usecase must orchestrate rather than own transition rules |
| Atomically persist mode/reason with boundaries | Session repository adapter | Storage schema/query changes | Domain must not know SQL columns |
| Classify host signals | Future #1436 finalization usecase/adapters | Host contracts change | This issue deliberately does not infer runtime evidence |
| Preserve portable session state | Bundle row/SQLite adapter | Bundle representation changes | Host adapters do not own portability |

### Boundaries / interfaces
| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| `RuntimeModeFrom` / `TerminalReasonFrom` | Domain/application/SQLite restore | Known string sets | Empty and unknown values return explicit errors |
| `NewSessionWithRuntimeMode` | Future one-shot finalizer | Session construction defaults | Invalid zero mode is rejected; existing `NewSession` remains explicit interactive compatibility |
| `Session.Terminate` | Future typed finalization usecase | First-terminal-state comparison | Returns applied/already_applied; conflicting reason wraps a typed conflict error and preserves current state |
| `SessionFromSnapshot(SessionSnapshot)` | SQLite and bundle adapters | Legacy fallback rules | Invalid active/terminal combinations fail closed |
| Migration 24 | Existing stores | Column addition/backfill | Existing fields remain unchanged; active reason stays empty; ended reason becomes legacy_unknown |

### Behavior tests
| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| Modes reject zero confusion | Empty/unknown, interactive, one_shot, resumed, background | Parse each value | Only the four explicit values succeed | Domain |
| First terminal wins | Active one-shot session | Terminate with success | State is applied with original time/reason/summary | Domain |
| Duplicate is idempotent | Success-terminal session | Terminate success again with different metadata | Returns already_applied and preserves original state | Domain |
| Conflict fails closed | Success-terminal session | Terminate failure | Typed conflict error; original state remains diagnosable | Domain |
| Migration is conservative | Pre-v24 active and ended rows | Initialize current schema | Both restore; modes are interactive; ended row reason is legacy_unknown; all old fields match | SQLite migration |
| Persistence is complete | Non-default modes and each reason | Save and restore | Mode/reason round-trip with one effective ended_at | SQLite integration |
| Bundle is portable | One-shot terminal session | Export/import | Mode and terminal reason survive; older rows without fields restore conservatively | Bundle integration |

### Risks / rollback
- Procedural-logic risk: do not branch on mode/reason inside handlers; the aggregate owns terminal comparison and typed values own validation.
- Premature-abstraction risk: use two concrete value objects and one transition result; do not introduce a generic state-machine framework.
- Migration/compatibility: additive columns only. Old binaries can still insert/start/end because defaults remain valid and no cross-column trigger blocks old updates. Current restore maps an ended row with an empty reason to `legacy_unknown`.
- Rollback: reverting the binary leaves additive columns ignored by older code. Reapplying the new binary restores typed state; no destructive table rewrite is required.
- Rollback trigger: any migration that changes existing timestamps/attribution, any path that overwrites a first terminal reason, or any zero value interpreted as one-shot requires immediate revert.

### Design checkpoint
- Risk is High because lifecycle state is persisted and exported.
- The migration is additive and conservative: unknown historical outcomes are named `legacy_unknown`, while every historical runtime defaults to interactive so no session becomes eligible for synthetic one-shot closure.
- The first terminal state is immutable. Same-reason retries are explicit no-ops; conflicts preserve and expose the stored reason rather than choosing a precedence rule in this foundational issue.
