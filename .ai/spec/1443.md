# Issue #1443: safe archive and backup capacity limits

## Structure-Behavior Design Note

### Requirement summary

- Purpose: bound local archive and SQLite-backup growth by age, count, and allocated bytes without deleting the only verified recovery point.
- Current state: archive creation/verification and SQLite backup/restore exist, but file roots have no read-only inventory, immutable cleanup plan, recovery floor, or crash-retry protocol.
- Expected behavior: a read-only plan lists exact regular-file identities, extents, reasons, verification state, and protected recovery floors. Hidden apply accepts only the reviewed plan ID, revalidates every identity, and deletes through durable same-directory tombstones.
- Non-goals: automatic/default-on cleanup, SQLite compaction, live-database replacement, external upload, or public apply before #1445 dogfood.

### Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| FileRetentionClass | `archive`, `backup` | selects verifier and independent budget | a file is never deleted through another class |
| FileIdentity | root ID, root-relative path, device/inode, size, mtime, SHA-256 | pins one exact regular file | symlinks, path escapes, and identity drift fail closed |
| CapacityBudget | optional age, count, allocated bytes | selects a deterministic union of violated ceilings | unknown or non-exclusive allocation makes the class indeterminate, never guessed |
| VerifiedRecoveryPoint | class, file identity, live generation, coverage, verification result | protects the newest recoverable file | each class needs a current-live-generation floor or that class is report-only |
| FileRetentionPlan | canonical payload, plan ID, ordered one-file batches | records reviewed deletion intent | planning performs no writes in database, roots, or sidecars |
| FileCleanupExecution | plan ID, candidate, state, tombstone name, sequence, fencing token | resumes deletion after interruption | only the state/name matrix below may advance; unknown combinations conflict |

### Deterministic capacity selection

Each class is evaluated independently against one configured root. Reserved Traceary journal, lease, tombstone, plan, and partial-file names are excluded from capacity inventory and reported separately.

1. Verify every regular candidate and derive `generation_created_at` from the archive manifest `created_at`; a catalog-recorded backup creation time is next; legacy backup filesystem `mtime` is the explicit last-resort provenance. The provenance kind is part of the canonical identity.
2. Order files by `generation_created_at`, generation digest, content digest, then root-relative path, oldest first. This is also the deletion order; ties never depend on directory iteration.
3. Select age violations where `generation_created_at < plan_time - max_age`.
4. For count, walk the same oldest-first order until the projected retained count is at most the ceiling. Protected, pinned, unverified, orphan, and otherwise ineligible files remain in the retained count; they are skipped for deletion but the deficit is not silently transferred past an unknown/blocking entry.
5. For allocated bytes, sum measured `st_blocks * 512` accounting bytes for all retained entries and select oldest eligible files until the projected sum is at most the ceiling. `allocated_bytes` is accounting consumption, not a guaranteed reclaim estimate. Any `nlink != 1`, reflink/clone status that cannot be proven exclusive, unsupported block accounting, overflow, or unknown extent makes this ceiling `indeterminate` and the class apply-ineligible.
6. A candidate is the set union of selections and records every applicable reason (`age`, `count`, `allocated_bytes`) in fixed enum order.
7. After the union, recompute all ceilings with every protected/blocking entry retained. If a ceiling remains exceeded because only protected, pinned, corrupt/unverified, orphaned, or otherwise ineligible entries remain, the class is `unsatisfiable`. `indeterminate` and `unsatisfiable` classes are report-only and have no apply batches.

Unverified/corrupt/orphan files count toward count/byte pressure but never become candidates and never cause a verified point to be selected merely to compensate. They therefore make an otherwise unreachable budget explicitly `unsatisfiable` rather than increasing destructive pressure.

### Recovery floor and apply-time guarantee

- Archive and backup classes each protect the newest fully verified point whose generation/coverage evidence matches the current live database generation supplied to the planner. A class with no such point is report-only.
- Archive verification includes format, manifest/payload digests, source generation, coverage, decryption availability, and copied-store rehearsal evidence. Backup verification includes exact digest, source generation, SQLite format, `integrity_check`, and copied-store representative-query evidence.
- The catalog owns `active/deleting/deleted`, pins, floor references, lease, and fencing token. Plan creation only reads it. Apply acquires the class lease and uses one catalog CAS to reserve `active -> deleting` only when pin count is zero and the point is not the current floor.
- Immediately before every namespace operation, apply re-verifies catalog identity, floor/pins, root identity, candidate identity, and verifier evidence. A changed floor, pin, generation, configuration, or candidate conflicts and requires a new plan.
- The lease/fencing token is held until namespace fsync, catalog/ledger update, and committed journal persistence complete. A stale token cannot resume or release another execution.

### Canonical execution payload

The hash-covered payload contains: schema/version; plan/expiry time; catalog/config identity; current live database generation; class and complete budgets; root ID plus device/inode identity; complete ordered inventory including exclusions/blockers; logical and allocated extents with provenance; candidate identities and every reason; one-file ordered batches; class result (`satisfied`, `unsatisfied`, `indeterminate`); floor generation/digest/coverage and verifier evidence digest; pin/floor snapshot; and the ordered execution steps. Execution-relevant arrays preserve canonical semantic order; JSON object key order alone is normalized. `display` is non-authoritative.

Apply parses and rehashes the plan, requires the exact confirmation ID, reconstructs all domain decisions from the canonical payload, and rejects any root, catalog/config, budget, inventory, floor, pin, verifier, batch, or identity drift. It never reads an execution decision from `display`.

### Root confinement and inventory completeness

- Planning and apply open the configured root directory without following symlinks, record root device/inode, and use directory-handle-relative operations for every path component. Intermediate and final symlinks are rejected; mount/device changes are blockers.
- Apply never turns a reviewed relative path back into an ambient absolute path. It opens the final regular file with no-follow semantics, compares device/inode/nlink/size/mtime/digest, and performs link/unlink through the held directory handles.
- Any unreadable entry, concurrent inventory change, symlink, hard link, unsupported extent, path escape, or unrecognized reserved-name collision marks that class indeterminate/apply-ineligible. A partial inventory never authorizes deletion of the remaining visible files.
- Tombstone creation uses no-replace semantics. Existing tombstones must match the exact reviewed identity and journal before recovery may use them; otherwise the execution conflicts.

### File deletion state/name matrix

Every journal write persists a complete record (`sequence`, plan/candidate identity, fencing token, state, original/tombstone names and identities), fsyncs the file, renames it over the journal, then fsyncs the directory. Namespace intent is durable before the operation. The tombstone is created with a no-replace directory-relative hard link to the exact open candidate, then the original is unlinked; initial `nlink` must be one. This avoids overwrite-prone rename while retaining a recoverable same-filesystem name.

| State | Original | Tombstone | Recovery action |
|---|---|---|---|
| `pending` | exact candidate | absent | acquire fenced lease, CAS catalog `active -> deleting`, persist `running` |
| `running` | exact candidate | absent | reverify all plan prerequisites and persist `tombstone_intent` |
| `tombstone_intent` | exact candidate | absent | no-replace link original to tombstone, fsync directory |
| `tombstone_intent` | exact candidate | same inode/digest | prior link completed; persist `tombstoned` |
| `tombstoned` | exact candidate | same inode/digest | persist `unlink_original_intent` |
| `unlink_original_intent` | exact candidate | same inode/digest | unlink original, fsync directory |
| `unlink_original_intent` | absent | exact tombstone | prior unlink completed; persist `original_unlinked` |
| `original_unlinked` | absent | exact tombstone | persist `unlink_tombstone_intent` |
| `unlink_tombstone_intent` | absent | exact tombstone | unlink tombstone, fsync directory |
| `unlink_tombstone_intent` | absent | absent | prior unlink completed; persist `tombstone_unlinked` |
| `tombstone_unlinked` | absent | absent | CAS catalog `deleting -> deleted`, append/fsync execution ledger, then persist `committed` |
| `committed` | absent | absent | terminal no-op |

At each row, an unknown identity/name combination, both names absent too early, a replacement original, a nonmatching tombstone, catalog drift, or stale fencing token persists/returns `conflicted` and performs no further deletion. `failed` is reserved for retryable I/O failure with the same recognized state/name combination; retry re-evaluates the matrix. Crash injection covers before/after every journal write, namespace operation, and directory fsync.

### Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| newest recovery floor | budgets select every file | plan | newest verified file is excluded and apply cannot delete it | application/integration |
| corrupt evidence | newest file corrupt, older file verified | plan | corrupt file is reported `unverified`; older verified floor remains protected | application |
| exact dry-run | mixed roots and budgets | plan | exact relative paths, digests, logical/allocated bytes, and all reasons are emitted; roots are byte/mtime unchanged | CLI/integration |
| symlink/path escape | symlink inside root | plan/apply | exclusion/conflict; outside target is never read or deleted | filesystem |
| identity drift | candidate replaced after planning | apply | conflict before tombstone | filesystem |
| interruption at every persistence boundary | any recognized matrix row | retry | no wrong deletion, zero/one effective deletion, terminal convergence | filesystem |
| restore floor | retained verified backup/archive | copied-store restore drill | SQLite/archive integrity and representative query pass | integration |
| orphan visibility | unverified pre-GC-looking backup | plan | it is reported but not deleted by default | CLI |

The test-vector table also covers every age/count/byte combination; exact ties; unknown/zero extents; pins; corrupt/orphan entries; protected-only unsatisfiable budgets; plan array/object mutation; root replacement; hard links; stale evidence/expiry; and OS block-accounting unavailability. Dry-run snapshots the database, WAL/SHM, root entry set/content/mtime, catalog, journal, markers, and temporary files; only the explicitly requested plan output is allowed to differ. Restore drills restore each class floor digest/current generation to a copy and require format verification, SQLite `integrity_check`, and representative metadata/full-body queries.

### Risks / rollback

- Procedural-risk control: the use case only orchestrates inventory, verification, policy, and executor ports; file deletion decisions live in tested value-based selection functions.
- Premature-abstraction control: support exactly two local classes and one root per class in v0.31; no provider/plugin system.
- Migration/compatibility: no database migration and no default behavior change. Apply remains hidden until #1445. Plan files are additive local artifacts.
- Rollback: apply never edits file content. Before unlink, the exact candidate receives a deterministic no-replace same-directory hard-link tombstone and the directory is synced. A committed deletion is recovered from the protected verified point, never from an unverified candidate.
- Plan freshness: plans expire after the operator-configured interval (default one hour); verifier/catalog evidence timestamps and expiry are hash-covered and rechecked at apply.
- OS extent support: Linux/Darwin use regular-file block accounting; unsupported platforms or unprovable clone exclusivity return `unknown_extent` and disable byte-budget apply for that class.
- Destructive implementation checkpoint: do not add unlink behavior until policy, dry-run purity, root confinement, identity-drift, and protected-floor tests pass.
