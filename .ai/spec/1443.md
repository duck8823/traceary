# Issue #1443: safe archive and backup capacity limits

## Structure-Behavior Design Note

### Requirement summary

- Purpose: bound local archive and SQLite-backup growth by age, count, and allocated bytes without deleting the only verified recovery point.
- Current state: archive creation/verification and SQLite backup/restore exist, but file roots have no read-only inventory, immutable cleanup plan, recovery floor, or crash-retry protocol.
- Expected behavior: a read-only plan lists exact regular-file identities, extents, reasons, verification state, and protected recovery floors. Hidden apply accepts only the reviewed plan ID, revalidates every identity, and deletes through durable same-directory tombstones.
- Non-goals: automatic/default-on cleanup, SQLite compaction, live-database replacement, external upload, or public apply before #1445 dogfood.

### Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| FileRetentionClass | `archive`, `backup` | selects verifier and independent budget | a file is never deleted through another class |
| FileIdentity | root ID, root-relative path, device/inode, size, mtime, SHA-256 | pins one exact regular file | symlinks, path escapes, and identity drift fail closed |
| CapacityBudget | optional age, count, allocated bytes | selects the union of violated ceilings | an unknown allocated extent makes the byte decision indeterminate, never guessed |
| VerifiedRecoveryPoint | class, file identity, generation, verification result | protects the newest recoverable file | at least one verified point per non-empty class remains |
| FileRetentionPlan | canonical payload, plan ID, ordered one-file batches | records reviewed deletion intent | planning performs no writes in database, roots, or sidecars |
| FileCleanupExecution | plan ID, candidate, state, tombstone name | resumes deletion after interruption | only `pending -> tombstoned -> committed`; unknown combinations conflict |

### Responsibility assignment

| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| budgets, recovery-floor selection, candidate reasons | application use case | policy decisions and reviewed intent | CLI/filesystem adapters must not choose deletions |
| root-confined inventory, file identity, allocated extent | filesystem adapter | OS metadata, symlink and path handling | application must not use `syscall.Stat_t` |
| archive/backup verification | small verifier ports/adapters | archive codec and SQLite integrity details differ | selector receives only verification evidence |
| tombstone state machine and durable journal | filesystem adapter | rename/unlink/fsync/crash recovery | CLI must not perform namespace operations |
| flags, plan file, exact plan-ID confirmation | hidden CLI commands | operator protocol | presentation must not reimplement policy |

### Boundaries / interfaces

| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| `RetentionFileInventory.List` | planner | directory handles, stat, digest, allocated blocks | escape/symlink/unreadable entries are reported as exclusions, not silently followed |
| `RetentionRecoveryVerifier.Verify` | planner/apply | archive manifest or SQLite integrity check | corrupt/unverified files never become a floor and never authorize deletion |
| `RetentionFileExecutor.Apply` | apply use case | tombstone journal, rename/unlink/fsync | stale identity or unknown crash state conflicts without wrong-path deletion |
| file-retention plan codec | hidden CLI | canonical hash and strict schema | unknown fields/order/hash mismatch fail closed |

### Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| newest recovery floor | budgets select every file | plan | newest verified file is excluded and apply cannot delete it | application/integration |
| corrupt evidence | newest file corrupt, older file verified | plan | corrupt file is reported `unverified`; older verified floor remains protected | application |
| exact dry-run | mixed roots and budgets | plan | exact relative paths, digests, logical/allocated bytes, and all reasons are emitted; roots are byte/mtime unchanged | CLI/integration |
| symlink/path escape | symlink inside root | plan/apply | exclusion/conflict; outside target is never read or deleted | filesystem |
| identity drift | candidate replaced after planning | apply | conflict before tombstone | filesystem |
| interruption after rename | matching tombstone remains | retry | digest-bound tombstone is unlinked and execution commits | filesystem |
| interruption before rename | original remains | retry | normal tombstone/unlink resumes | filesystem |
| restore floor | retained verified backup/archive | copied-store restore drill | SQLite/archive integrity and representative query pass | integration |
| orphan visibility | unverified pre-GC-looking backup | plan | it is reported but not deleted by default | CLI |

### TDD plan

| Behavior | Red | Green | Refactor target |
|---|---|---|---|
| policy selection | every-file budget deletes recovery floor | value-based selector protects newest verified point | keep ceiling decisions out of CLI |
| confined inventory | symlink/path escape is followed | directory-rooted lstat/digest inventory | isolate OS metadata adapter |
| immutable plan | tampering/order changes execute | strict canonical codec and exact identity conversion | share generic canonical JSON only |
| crash retry | tombstone is orphaned or wrong file removed | durable journal and state/name matrix | keep namespace state machine in adapter |

### Risks / rollback

- Procedural-risk control: the use case only orchestrates inventory, verification, policy, and executor ports; file deletion decisions live in tested value-based selection functions.
- Premature-abstraction control: support exactly two local classes and one root per class in v0.31; no provider/plugin system.
- Migration/compatibility: no database migration and no default behavior change. Apply remains hidden until #1445. Plan files are additive local artifacts.
- Rollback: apply never edits file content. Before unlink, the exact candidate is renamed to a deterministic same-directory tombstone and directory-synced. A committed deletion is recovered from the protected verified point, never from an unverified candidate.
- Destructive implementation checkpoint: do not add unlink behavior until policy, dry-run purity, root confinement, identity-drift, and protected-floor tests pass.
