# Issue #1443: safe archive and backup capacity limits

## Structure-Behavior Design Note

### Requirement summary

- Purpose: bound local archive and SQLite-backup growth by age, count, and allocated bytes without deleting the only verified recovery point.
- Current state: archive creation/verification and SQLite backup/restore exist, but file roots have no read-only inventory, immutable cleanup plan, recovery floor, or crash-retry protocol.
- Expected behavior: a read-only plan lists exact regular-file identities, extents, reasons, verification state, and protected recovery floors. Hidden apply accepts only the reviewed plan ID, revalidates every identity, and deletes through durable same-directory tombstones.
- Non-goals: automatic/default-on cleanup, SQLite compaction, live-database replacement, external upload, or public apply before #1445 dogfood.

### Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| FileRetentionClass | `archive`, `backup` | selects verifier and independent budget | a file is never deleted through another class |
| FileIdentity | root ID, root-relative path, device/inode, size, mtime, SHA-256 | pins one exact regular file | symlinks, path escapes, and identity drift fail closed |
| CapacityBudget | optional age, count, allocated bytes | selects a deterministic union of violated ceilings | unknown or non-exclusive allocation makes the class indeterminate, never guessed |
| VerifiedRecoveryPoint | class, file identity, live generation, coverage, verification result | protects the newest recoverable file | each class needs a current-live-generation floor or that class is report-only |
| FileRetentionPlan | canonical payload, plan ID, ordered one-file batches | records reviewed deletion intent | planning performs no writes in database, roots, or sidecars |
| FileCleanupExecution | plan ID, candidate, state, tombstone name, sequence, fencing token | resumes deletion after interruption | only the state/name matrix below may advance; unknown combinations conflict |

### Deterministic capacity selection

Each class is evaluated independently against one configured root. Reserved Traceary journal, lease, tombstone, plan, and partial-file names are excluded from capacity inventory and reported separately.

1. Inventory every entry. An unreadable entry, symlink, hard link, unknown extent required by an active byte ceiling, unknown reserved-name collision, or unrecognized file is a blocker. A blocker makes the class `indeterminate`; the canonical inventory still reports all evidence collected, but candidates and batches are empty.
2. Verify every regular candidate and derive `generation_created_at` from the archive manifest `created_at`; a catalog-recorded backup creation time is next; legacy backup filesystem `mtime` is the explicit last-resort provenance. The provenance kind is part of the canonical identity.
3. Order files by `generation_created_at`, generation digest, content digest, then root-relative path, oldest first. This is also the deletion order; ties never depend on directory iteration.
4. Compute each configured ceiling independently against the original complete known inventory. Age selects every eligible entry strictly older than `plan_time - max_age`.
5. Count starts with the original retained count and walks the full oldest-first order until the projected retained count is within the ceiling. An ineligible known entry is skipped without decrementing the projection, and the walk continues to later eligible entries. An unknown/blocking entry already made the class `indeterminate` in step 1, so selection never guesses past incomplete evidence.
6. Allocated bytes follows the same rule: start with the original measured `st_blocks * 512` sum and walk the full order, subtracting only independently eligible selected entries until within the ceiling. `allocated_bytes` is accounting consumption, not a guaranteed reclaim estimate. Overflow or unknown/non-exclusive allocation makes the class `indeterminate` before selection.
7. Take the set union of the three independent selections. Every candidate records every ceiling that selected it (`age`, `count`, `allocated_bytes`) in that fixed enum order, even when another ceiling would already remove it.
8. Recompute every ceiling after the union, retaining the floor, pins, corrupt/unverified/orphaned entries, and all other ineligible known entries. The only canonical results are `satisfied`, `unsatisfied`, and `indeterminate`. If complete known evidence proves a configured ceiling cannot be reached, result is `unsatisfied`. If evidence is incomplete or allocation is unknown, result is `indeterminate`. Either result clears all candidates and batches; only `satisfied` may carry one-candidate batches.

Unverified/corrupt/orphan files count toward known count/byte pressure but never become candidates and never cause a verified point to be selected merely to compensate. Complete known evidence can therefore produce `unsatisfied`; incomplete evidence produces `indeterminate`. Neither increases destructive pressure.

### Recovery floor and apply-time guarantee

- Archive and backup classes each protect the newest fully verified point whose generation/coverage evidence matches the current live database generation supplied to the planner. A class with no such point is report-only.
- Archive verification includes format, manifest/payload digests, source generation, coverage, decryption availability, and copied-store rehearsal evidence. Backup verification includes exact digest, source generation, SQLite format, `integrity_check`, and copied-store representative-query evidence.
- The catalog owns `active/deleting/deleted`, pins, floor references, lease, and fencing token. Plan creation only reads it. Apply acquires the class lease and uses one catalog CAS to reserve `active -> deleting` only when pin count is zero and the point is not the current floor.
- Immediately before every namespace operation, apply re-verifies catalog identity, floor/pins, root identity, candidate identity, and verifier evidence. A changed floor, pin, generation, configuration, or candidate conflicts and requires a new plan.
- The lease/fencing token is held until namespace fsync, catalog/ledger update, and committed journal persistence complete. A stale token cannot resume or release another execution.

### Canonical execution payload

The hash-covered payload contains: schema/version; plan/expiry time; catalog/config identity; current live database generation; class and complete budgets; root ID plus device/inode identity; complete ordered inventory including exclusions/blockers; logical and allocated extents with provenance; candidate identities and every reason; one-file ordered batches; class result (`satisfied`, `unsatisfied`, `indeterminate`); floor generation/digest/coverage and verifier evidence digest; pin/floor snapshot; and the ordered execution steps. Execution-relevant arrays preserve canonical semantic order; JSON object key order alone is normalized. `display` is non-authoritative.

Apply parses and rehashes the plan, requires the exact confirmation ID, reconstructs all domain decisions from the canonical payload, and rejects any root, catalog/config, budget, inventory, floor, pin, verifier, batch, or identity drift. It never reads an execution decision from `display`.

### Root confinement and inventory completeness

- Planning and apply open the configured root directory without following symlinks, record root device/inode, and use directory-handle-relative operations for every path component. Intermediate and final symlinks are rejected. Every descendant directory and file must have the root's device ID; a root-internal mount point or any device-boundary change is a blocker rather than a second allowed filesystem.
- Apply never turns a reviewed relative path back into an ambient absolute path. It opens the final regular file with no-follow semantics, compares device/inode/nlink/size/mtime/digest, and performs link/unlink through the held directory handles.
- Any unreadable entry, concurrent inventory change, symlink, hard link, unsupported extent, path escape, or unrecognized reserved-name collision marks that class indeterminate/apply-ineligible. A partial inventory never authorizes deletion of the remaining visible files.
- Tombstone creation uses no-replace semantics. Existing tombstones must match the exact reviewed identity and journal before recovery may use them; otherwise the execution conflicts.

### File deletion state/name/catalog/ledger matrix

Every journal write persists a complete record (`sequence`, plan/candidate identity, fencing token, state, original/tombstone names and identities), fsyncs the file, renames it over the journal, then fsyncs the directory. Namespace intent is durable before the operation. The original is atomically moved with a no-replace directory-relative rename to a reserved tombstone; the moved identity and digest are then verified before unlink. If an ordinary concurrent writer replaces the original between verification and rename, post-rename verification fails and the unexpected file is atomically restored whenever the original name is free; it is never unlinked. Initial `nlink` must be one. Apply requires a caller-owned root with no group/other write permission. Traceary writers share the root lock and never mutate another execution's reserved tombstone name. A malicious process with the caller's same UID is explicitly outside the threat model: portable Unix exposes name-based unlink but no atomic conditional unlink by verified inode, and that process can already modify the caller's Traceary database and configuration.

`token` below means the exact current fenced execution token. The ledger has an idempotent unique key `(plan_id, candidate_id, file_identity)`; a nonmatching row is always a conflict.

The root also has a reserved, directory-handle-opened advisory lock file. An executor must hold its exclusive kernel lock continuously from before token acquisition/takeover until after `committed` is durable and the catalog lease is released. A paused prior process therefore retains the kernel lock and cannot be fenced; a crashed process loses it before a successor can proceed. Token checks alone never protect a namespace syscall.

Every successful resume issues a strictly increasing token after acquiring that kernel lock. For a first execution, catalog CAS installs the new token. For a crash resume, the successor requires an expired catalog lease, the exact prior journal token/sequence/hash, and the exact catalog item/ledger row described by the matrix. It then performs one durable catalog CAS from `lease(old)` to `lease(new, takeover_from=old, journal_sequence, journal_hash)` without changing the item state. Only after that CAS does it persist the same journal state and sequence with `token=new`, then clear `takeover_from` by catalog CAS. If it crashes after the first CAS, a later lock holder recognizes the exact catalog takeover record plus old journal and completes the journal token update; if it crashes after the journal update, the exact new token/hash permits clearing the takeover record. Any other old/new token, sequence, hash, unexpired lease, catalog item, or ledger combination conflicts. Once the kernel lock is held, the executor rechecks the catalog token immediately before every namespace operation; the lock remains held across that operation and its directory fsync.

Token-takeover crash vectors are mandatory as a parameterized cross-product with every nonterminal normal-matrix row from `pending/active/ledger-absent` through `catalog_deleted/deleted(plan)/ledger-exact`. At the injected crash, the catalog item and ledger remain exactly those of the selected base row; takeover changes only the lease record and journal token. Every cross-product case then resumes the base row to the same concrete terminal result.

| Injection point | Durable lease / journal; item and ledger equal base row | Retry action | Final journal / catalog item / ledger rows / logical deletions |
|---|---|---|---|
| after kernel lock, before takeover CAS | `lease(old)` / journal old | issue new token by exact CAS | committed / deleted(plan) / 1 / 1 |
| after takeover CAS, before journal write | `lease(new,takeover_from=old,seq,hash)` / journal old | verify tuple, persist same state with new token | committed / deleted(plan) / 1 / 1 |
| after journal write, before clearing takeover | `lease(new,takeover_from=old,seq,hash)` / journal new | verify tuple, clear takeover marker | committed / deleted(plan) / 1 / 1 |
| after clearing takeover | `lease(new)` / journal new | enter the selected base matrix row | committed / deleted(plan) / 1 / 1 |

| Journal state | Original | Tombstone | Catalog | Ledger | Recovery action |
|---|---|---|---|---|---|
| `pending` | exact | absent | `active` | absent | acquire lease/token, CAS `active -> deleting(token)`, then persist `running` |
| `pending` | exact | absent | `deleting(token)` | absent | crash followed catalog CAS; persist `running` |
| `running` | exact | absent | `deleting(token)` | absent | reverify all prerequisites; persist `tombstone_intent` |
| `tombstone_intent` | exact | absent | `deleting(token)` | absent | no-replace rename original to tombstone, verify moved identity, fsync directory |
| `tombstone_intent` | absent | same identity | `deleting(token)` | absent | prior rename completed; persist `tombstoned` |
| `tombstoned` | absent | same identity | `deleting(token)` | absent | persist `unlink_original_intent` |
| `unlink_original_intent` | absent | exact | `deleting(token)` | absent | confirm the atomic move left the original absent; persist `original_unlinked` |
| `original_unlinked` | absent | exact | `deleting(token)` | absent | persist `unlink_tombstone_intent` |
| `unlink_tombstone_intent` | absent | exact | `deleting(token)` | absent | unlink tombstone, fsync directory |
| `unlink_tombstone_intent` | absent | absent | `deleting(token)` | absent | prior unlink completed; persist `tombstone_unlinked` |
| `tombstone_unlinked` | absent | absent | `deleting(token)` | absent | persist `catalog_commit_intent` |
| `catalog_commit_intent` | absent | absent | `deleting(token)` | absent | append and fsync the unique ledger record, then persist `ledger_recorded` |
| `catalog_commit_intent` | absent | absent | `deleting(token)` | exact | crash followed ledger append; persist `ledger_recorded` |
| `ledger_recorded` | absent | absent | `deleting(token)` | exact | CAS `deleting(token) -> deleted(plan, token)`, fsync catalog, persist `catalog_deleted` |
| `ledger_recorded` | absent | absent | `deleted(plan, token)` | exact | crash followed catalog CAS; persist `catalog_deleted` |
| `catalog_deleted` | absent | absent | `deleted(plan, token)` | exact | persist `committed` |
| `committed` | absent | absent | `deleted(plan, token)` | exact | terminal no-op; release lease only after committed journal fsync |

At each row, an unknown identity/name combination, both names absent too early, a replacement original, a nonmatching tombstone, catalog state or owner-token drift, a nonmatching ledger record, or stale fencing token persists/returns `conflicted` and performs no further deletion. The complete state enum is `pending`, `running`, `tombstone_intent`, `tombstoned`, `unlink_original_intent`, `original_unlinked`, `unlink_tombstone_intent`, `tombstone_unlinked`, `catalog_commit_intent`, `ledger_recorded`, `catalog_deleted`, `committed`, `failed`, and `conflicted`. `failed` is reserved for retryable I/O failure with the same recognized row; retry re-evaluates the matrix. Crash injection covers before/after every journal write, namespace operation, directory fsync, ledger append, and catalog CAS. A lease/token is never released before `committed` is durable.

### Deterministic selection test vectors

Unless overridden, inventory is three verified current-generation entries: `A=(2026-01-01T00:00Z,10 blocks,digest-a,a.arc)`, `B=(2026-01-02T00:00Z,10 blocks,digest-b,b.arc)`, and protected floor `C=(2026-01-03T00:00Z,10 blocks,digest-c,c.arc)`; one block is the test adapter's accounting unit. Plan time is `2026-01-10T00:00Z`.

All selection vectors are plan-only. Their exact postcondition is therefore `journal=absent`, catalog items unchanged at `active`, `ledger_rows=0`, and `logical_candidate_deletions=0`.

| Vector | Override / budget | Expected ordered candidates and reasons | Result / batches | Journal / catalog / ledger / deletions |
|---|---|---|---|---|
| age | `max_age=8d` | `A:[age]` | `satisfied`; one batch for A | absent / active / 0 / 0 |
| count | `max_count=2` | `A:[count]` | `satisfied`; one batch for A | absent / active / 0 / 0 |
| bytes | `max_allocated=20` | `A:[allocated_bytes]` | `satisfied`; one batch for A | absent / active / 0 / 0 |
| composite | all three ceilings above | `A:[age,count,allocated_bytes]` | `satisfied`; one batch for A | absent / active / 0 / 0 |
| skip known pin | pin A, `max_count=2` | `B:[count]` | `satisfied`; one batch for B | absent / active / 0 / 0 |
| protected-only deficit | pin A, floor C, `max_count=1` | none after fail-closed clearing | `unsatisfied`; no batches | absent / active / 0 / 0 |
| unknown extent | B allocated extent unknown, byte ceiling active | none | `indeterminate`; no batches | absent / active / 0 / 0 |
| corrupt/orphan pressure | A corrupt, B orphan, floor C, `max_count=1` | none | `unsatisfied`; no batches | absent / active / 0 / 0 |
| incomplete inventory | unreadable `x.arc` plus normal inventory | none | `indeterminate`; no batches | absent / active / 0 / 0 |
| exact tie | A and B same timestamp/generation/content digest; paths `a.arc`,`b.arc`; `max_count=2` | `A:[count]` by relative path | `satisfied`; one batch for A | absent / active / 0 / 0 |
| zero bytes | all extents zero, `max_allocated=0` | none | `satisfied`; no batches | absent / active / 0 / 0 |

### Crash-injection test vectors

Every vector retries twice and asserts no outside-root access, no wrong-identity deletion, exactly zero or one matching ledger row, and convergence to `committed` when the recognized row remains valid.

| Injection point | Durable state seen on retry | Expected recovery | Final journal / catalog / ledger rows / logical deletions |
|---|---|---|---|
| after `active -> deleting(token)` before `running` | `pending`, catalog deleting(token), ledger absent | fenced takeover, persist `running`, continue once | committed / deleted(plan) / 1 / 1 |
| after tombstone intent before rename | `tombstone_intent`, original exact, tombstone absent | fenced takeover, rename once | committed / deleted(plan) / 1 / 1 |
| after rename/fsync before state | `tombstone_intent`, original absent, tombstone exact | fenced takeover, persist `tombstoned`, do not rename again | committed / deleted(plan) / 1 / 1 |
| after original unlink/fsync before state | `unlink_original_intent`, original absent, tombstone exact | fenced takeover, persist `original_unlinked` | committed / deleted(plan) / 1 / 1 |
| after tombstone unlink/fsync before state | `unlink_tombstone_intent`, both absent | fenced takeover, persist `tombstone_unlinked` | committed / deleted(plan) / 1 / 1 |
| after ledger append/fsync before `ledger_recorded` | `catalog_commit_intent`, exact ledger present | fenced takeover, persist `ledger_recorded`, never append duplicate | committed / deleted(plan) / 1 / 1 |
| after catalog delete/fsync before `catalog_deleted` | `ledger_recorded`, catalog deleted(plan), exact ledger | fenced takeover, persist `catalog_deleted` | committed / deleted(plan) / 1 / 1 |
| after committed journal fsync before lease release | `committed`, catalog/ledger exact | fenced takeover only to release exact lease; no namespace operation | committed / deleted(plan) / 1 / 1 |

### Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| newest recovery floor | budgets select every file | plan | newest verified file is excluded and apply cannot delete it | application/integration |
| corrupt evidence | newest file corrupt, older file verified | plan | corrupt file is reported `unverified`; older verified floor remains protected | application |
| exact dry-run | mixed roots and budgets | plan | exact relative paths, digests, logical/allocated bytes, and all reasons are emitted; roots are byte/mtime unchanged | CLI/integration |
| symlink/path escape | symlink inside root | plan/apply | exclusion/conflict; outside target is never read or deleted | filesystem |
| identity drift | candidate replaced after planning | apply | conflict before tombstone | filesystem |
| interruption at every persistence boundary | any recognized matrix row | retry | no wrong deletion, zero/one effective deletion, terminal convergence | filesystem |
| restore floor | retained verified backup/archive | copied-store restore drill | SQLite/archive integrity and representative query pass | integration |
| orphan visibility | unverified pre-GC-looking backup | plan | it is reported but not deleted by default | CLI |

The tabulated vectors are binary acceptance tests, not examples: candidate order, reason arrays, result enum, batch count, catalog state, and ledger cardinality must match exactly. Additional binary vectors cover plan array/object mutation, root replacement, hard links, stale evidence/expiry, and OS block-accounting unavailability. Dry-run snapshots the database, WAL/SHM, root entry set/content/mtime, catalog, journal, markers, and temporary files; only the explicitly requested plan output is allowed to differ. Restore drills restore each class floor digest/current generation to a copy and require format verification, SQLite `integrity_check`, and representative metadata/full-body queries.

### Risks / rollback

- Procedural-risk control: the use case only orchestrates inventory, verification, policy, and executor ports; file deletion decisions live in tested value-based selection functions.
- Premature-abstraction control: support exactly two local classes and one root per class in v0.31; no provider/plugin system.
- Migration/compatibility: no database migration and no default behavior change. Apply remains hidden until #1445. Plan files are additive local artifacts.
- Rollback: apply never edits file content. Before unlink, the exact candidate is atomically moved to a deterministic no-replace same-directory tombstone, reverified, and the directory is synced. A committed deletion is recovered from the protected verified point, never from an unverified candidate.
- Plan freshness: plans expire after the operator-configured interval (default one hour); verifier/catalog evidence timestamps and expiry are hash-covered and rechecked at apply.
- OS extent support: Linux/Darwin use regular-file block accounting; unsupported platforms or unprovable clone exclusivity return `unknown_extent` and disable byte-budget apply for that class.
- Destructive implementation checkpoint: do not add unlink behavior until policy, dry-run purity, root confinement, identity-drift, and protected-floor tests pass.
