# #1206 — Read surfaces should not amplify large tool payloads into agent context

## Problem

`traceary sessions --snapshot --json` (and `top --snapshot --json`) re-emits the
session tree's `latest_event_message` verbatim. A noisy `command_executed`
payload (e.g. `traceary doctor --json`, an MCP payload dump) lands in the node
unbounded and is fed straight back into the next agent's context. The recent
failures / recent commands panes already cap bodies via
`newTruncatedEventOutput` + `DefaultTopSnapshotBodyLimit`, but the session node
`latest_event_message` does not. `reliability.large_payloads` reports counts but
no investigable, body-safe metadata.

## Scope (read-surface only)

- No DB schema or migration changes. Stored event bodies are preserved.
- Ingestion truncation unchanged. `command_executed` storage untouched.
- `traceary show` / explicit event detail keep full body by default.

## Design

### 1. Truncate snapshot node `latest_event_message`

Apply the shared `apptypes.TruncateCommandPayload` policy with
`DefaultTopSnapshotBodyLimit` to `topSnapshotNode.LatestEventMessage`, matching
the failures/recent-commands panes. Additive metadata announces the cut:

- `latest_event_message_truncated` (omitempty)
- `latest_event_message_length` (rune count, omitempty)
- `latest_event_message_bytes` (byte count, omitempty)

Existing `latest_event_message` / `latest_event_kind` / `latest_event_at` field
names are unchanged.

### 2. Retrieval hint: `latest_event_id`

Add `latest_event_id` to `SessionSummary` (sourced from the existing
`latest_events` CTE via the events table `id` column — query-only, no schema
change) and surface it on `topSnapshotNode` as `latest_event_id`. Consumers
fetch the full body explicitly with `traceary show <latest_event_id>`.

### 3. `large_payloads.samples`

Extend `reliability.large_payloads` with a bounded `samples` array. Each sample
is metadata-only (never the full body):

- `event_id`, `kind`, `source` (`failure` | `recent_command`)
- `message_length` (runes), `message_bytes`
- `first_line` — first body line, whitespace-collapsed and rune-bounded
- `retrieval_hint` — `traceary show <event_id>`

Samples are deduped by event id (an event appearing in both failures and recent
commands yields one row) and capped at `topLargePayloadSampleLimit`.

### 4. Self-diagnostic / MCP budget (already satisfied — regression-guarded)

- Hook audit suppression of Traceary self-inspection commands already exists
  (`isTracearySelfInspectionCommand`). Untouched.
- MCP `get_context` / `search` / `list_events` already default to
  `DefaultListEventBodyLimit` via `resolveBodyLimit`. `session_status`
  lineage/tree/handoff already omit latest_event fields. Add regression tests.

## Behavior tests

- snapshot node `latest_event_message` truncates + exposes id/metadata; full
  body retrievable via `traceary show <latest_event_id>`.
- `large_payloads.samples` are metadata-only, body-safe, deduped.
- MCP `get_context` default body_limit stays capped; `session_status` does not
  leak latest_event fields.
