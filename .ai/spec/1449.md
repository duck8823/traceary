# Issue #1449: snapshot-consistent usage aggregates

## Structure-Behavior Design Note

### Requirement summary

- **Goal:** Add one body-free usage aggregate to the existing report snapshot so
  the CLI and MCP expose identical totals and completeness metadata.
- **Current state:** finalized usage observations and immutable run attribution
  exist, but no read model selects the current snapshot head or prevents
  excluded alternatives from contributing to totals.
- **Expected behavior:** one SQLite read transaction loads report sources and
  usage rows; the application groups additive/current-snapshot rows and exposes
  excluded/unavailable coverage without estimating missing values.
- **Non-goals:** infer role, round, wall time, provider billing, retry count, or
  private prompt/transcript content that is not durably recorded.

### Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Usage report row | finalized observation + optional run facts | contributes coverage; contributes totals only when accounting-eligible | pending rows and superseded snapshots are absent |
| Aggregate key | provider, engine, model, role, repository, ticket, PR, batch, round | groups comparable observations | unavailable dimensions are explicit and never fabricated |
| Metric coverage | known/unavailable counts + optional sum | distinguishes zero from missing | excluded evidence never contributes to sums |
| Run fact | run identity + packet/tool bytes | contributes once per run per aggregate key | multiple observations for one run cannot multiply run bytes |
| Cost total | currency + origin | separates provider-reported from estimated | estimated is never labelled provider-reported |
| Snapshot extent | returned row count and result-cap provenance | marks complete/partial report | CLI and MCP share the same snapshot DTO |

### Responsibility assignment

| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| Snapshot transaction, latest-snapshot selection, body-free row scan | SQLite report datasource | storage/query semantics | usecase must not contain SQL |
| Accounting eligibility, grouping, metric/run dedupe | report usecase | provider-neutral report behavior | CLI/MCP must not duplicate rules |
| Validated result schema and availability shapes | application types | shared CLI/MCP contract | infrastructure DTOs must not leak |
| Text rendering | CLI | presentation only | no aggregation decisions |
| JSON tool exposure | MCP | protocol registration only | same usecase/result as CLI |

### Boundaries / interfaces

| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| `ReportQueryService.LoadReportWindow` | report usecase | one SQLite read transaction and paging | invalid stored enum/timestamp fails closed |
| `ReportWindow.Usage` | report usecase | SQL row/null representation | only body-free normalized records |
| `ReportUsecase.Generate` | CLI/MCP | grouping maps and run dedupe | invalid extent or unknown finalized state fails closed |
| `ReportSnapshot.Usage` | CLI/MCP clients | storage layout | explicit complete/partial and unavailable fields |

### Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| Snapshot head only | two revisions in one snapshot series | report loads rows | only the unsuperseded revision is present | SQLite |
| No duplicate accounting | additive, excluded alternative, repeated run attribution | report aggregates | tokens count once; excluded coverage remains visible; run bytes count once | usecase |
| Availability preserved | known zero and unavailable values | report aggregates | zero has known count/value 0; unavailable increments separately | usecase |
| Cost truth preserved | estimated and provider-reported costs | report aggregates | separate origin-labelled totals | usecase |
| Result cap explicit | more usage rows than cap | report loads | usage extent is partial with `result_cap` | SQLite/usecase |
| Shared surface | same report usecase result | CLI JSON and MCP tool return | usage schema and totals are identical | presentation |
| Private-body independence | events contain bodies but usage query selects metadata tables only | report loads | no body column is queried or returned | SQLite review/test |

### TDD plan

### Risks / rollback

- **Procedural risk:** avoid putting counter decisions in CLI/MCP or SQL string
  concatenation; application aggregation owns them.
- **Premature abstraction:** reuse the existing report snapshot instead of a
  second reporting stack; add no provider-specific strategies.
- **Migration / compatibility:** no schema migration. New JSON fields are
  additive. Existing finalized rows remain readable.
- **Unavailable dimensions:** role, round, and wall time are not currently
  persisted. The report emits them as unavailable rather than deriving them
  from agent names, parent depth, or timestamps.
- **Rollback trigger:** any token double count, superseded snapshot inclusion,
  provider-reported/estimated cost conflation, or CLI/MCP divergence.
- **Rollback:** revert the report projection and surface additions; no durable
  data rollback is required.
- **PR plan:** one issue/one PR. Commit 1 adds read model and aggregation;
  commit 2 adds shared CLI/MCP presentation and documentation if separation is
  useful without breaking intermediate builds.

### Design checkpoint

Proceed with the migration-free extension of the existing report snapshot.
This keeps snapshot consistency and shared CLI/MCP semantics in one boundary,
and it makes currently unrecorded dimensions explicitly unavailable rather
than inventing data.
