# Structure-Behavior Design Note: cross-host delivery identity

## Requirement summary

- Purpose: make an exact supported-hook redelivery store one logical event while preserving legitimate repeated commands and prompts.
- Current behavior: SQLite suppresses hook content and command events by body equality inside a two-second window; session start has a separate presentation-layer no-op path.
- Expected behavior: use a host-native, session- and hook-scoped delivery identity when one exists; keep canonical session workspace immutable and resolve event workspace from event-local evidence before session fallback.
- Non-goals: heuristic historical deletion, workspace-conflict reporting UI, body-only identity, and one-shot lifecycle changes.

The normative data and rollback contract is `docs/operations/workspace-identity-contract.md`.

## Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Hook delivery input | native delivery ID, raw workspace | presentation extracts supported host fields | no body hash fallback when the host has no stable ID |
| Delivery evidence | reported identity, delivery fingerprint, attribution fingerprint | application derives normalized hashes | delivery fingerprint excludes workspace; attribution fingerprint includes it |
| Delivery ledger | accepted or conflict row, observed event ID | repository resolves exact retry and identity collision | one accepted fingerprint per reported identity; one row per fingerprint |
| Workspace observation | primary or supplemental, runtime or backfill | records effective workspace provenance | primary observation is atomic with a new event; retry never rewrites the event |
| Session workspace | canonical workspace | fixed by the first successful start | event delivery cannot mutate it |
| Event workspace | effective workspace | event-local evidence, then canonical fallback | immutable after event insert |

## Responsibility assignment

| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| native ID and raw cwd extraction | host adapter in `presentation/cli` | host payload versions change | domain must not know JSON field names |
| identity namespace and fingerprints | application/domain value construction | semantic identity contract changes | handler must not decide deduplication |
| exact/conflict decision and atomic writes | event/session repository implementation | transaction/schema behavior changes | usecase must not contain SQL or retry loops |
| workspace relationship | domain pure rule | canonical/effective rule changes | SQLite must not own path semantics |
| catch-up paging and write ordering | SQLite infrastructure | batch/locking changes | domain/application must not know table details |

## Boundaries / interfaces

| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| hook-delivery context input | event/session usecases | host JSON paths | absent stable ID means unconditional distinct delivery |
| event delivery evidence | repositories | hashing input representation | invalid/empty evidence is rejected before persistence |
| existing `EventRepository` save methods | usecases | ledger/observation transactions | identity retry is success; schema/I/O/busy/check errors fail |
| store initialization | CLI/MCP composition | migration and catch-up batches | schema completion permits open; incomplete catch-up is diagnostic only |

## Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| exact redelivery | same session/hook/native ID and delivery fingerprint | saved twice | one event and one primary observation | SQLite integration |
| changed retry attribution | same delivery, new cwd | saved again | original event workspace unchanged; one supplemental observation | SQLite integration |
| identity conflict | same reported ID, different delivery fingerprint | saved and retried | one accepted event and one conflict event, with no amplification | SQLite integration |
| legitimate repeat | no stable native ID, equal body | saved twice | two events | application + SQLite |
| session isolation | equal native ID/body in different sessions | saved | two events | SQLite integration |
| host identity | representative Codex, Claude, Antigravity, Gemini, Grok, Kimi payloads | normalized | only proven native identifiers create delivery evidence | presentation fixtures |
| workspace precedence | canonical A and event-local B | hook event saved | session remains A and event is B | CLI integration |
| migration catch-up | legacy events without observations | initialize repeatedly | primary backfill observations converge without blocking | migration integration |

## Risks / rollback

- Procedural risk: duplicating the ledger algorithm across log, audit, and boundary paths. Use one SQLite transaction helper with optional audit/session work.
- Premature abstraction risk: do not create a host plugin strategy; field extraction remains small adapter functions.
- Migration/compatibility: add tables/indexes only; keep `sessions.workspace`, `events.workspace`, IDs, and default filters unchanged. Catch-up uses bounded batches and primary backfill observations.
- Rollback: deploy the prior binary while retaining additive tables. Never drop runtime provenance as part of rollback.
- Rollback trigger: existing row/ID changes, body-only suppression, default-filter widening, normal-ingest blocking, or inability to replay an exact delivery idempotently.

## Implementation split inside the single issue/PR

1. schema and domain evidence;
