## Structure-Behavior Design Note

### Requirement summary
- Purpose: keep routine workspace-identity reporting body-free and bounded while retaining an explicit, diagnosable historical content heuristic.
- Current state: every report invokes `DedupeContentEvents`, materializing all eligible prompt/transcript bodies even when the operator only needs exact delivery and workspace metrics.
- Expected behavior: the default report never invokes body-based dedupe; explicit heuristic measurement scans at most a configured row limit and reports `not_requested`, `partial`, `complete`, or `failed` independently from exact delivery.
- Out of scope: changing the stable-ID ledger, replacing the historical heuristic with a persisted hash projection, or changing apply/restore cleanup semantics.

### Conceptual model
| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Exact identity report | workspace observations and runtime attempt ledger | Always builds body-free metrics | Never depends on heuristic success or request state |
| Heuristic measurement request | not requested or requested with row limit and strictness | Optionally invokes dry-run content dedupe | Report path never applies or restores rows |
| Heuristic measurement | not_requested, partial, complete, failed | Describes whether the requested population was measured | State is explicit in JSON even when counts are zero |
| Bounded dedupe sample | scanned rows and total eligible rows | Loads at most `MaxScanRows` bodies | `Apply` cannot be combined with a bound |

### Responsibility assignment
| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| Interpret report flags and render measurement state | CLI report adapter | Operator contract and output format change | Store; it should not know CLI state names |
| Validate bounded dry-run versus apply | Store-management use case | Maintenance safety rule changes | SQLite adapter; application rule must be shared by callers |
| Count eligible rows and bound body loading | SQLite dedupe datasource | Query strategy or schema changes | CLI; SQL and body materialization stay in infrastructure |
| Preserve cleanup behavior | Existing store dedupe command | Apply/restore contract changes | Workspace report; it remains dry-run only |

### Boundaries / interfaces
| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| `report workspace-identity` | Operator/automation | Body-free exact queries | Existing store required; default succeeds without dedupe dependency |
| `--include-heuristic --heuristic-limit N` | Operator/automation | Bounded content scan | Positive bound; measurement failure still emits exact JSON with `failed`, then returns non-zero |
| `ContentEventDedupeParams.MaxScanRows` | Report use case path | SQL LIMIT implementation | Negative rejected; apply plus bound rejected |
| `ContentEventDedupeResult.TotalEligibleCount` | Report state builder | Count query | `scanned < total` means partial; otherwise complete |

### Behavior tests
| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| Default is body-free | Exact report dependencies only | Command runs without heuristic flags | Dedupe is not called and state is `not_requested` | CLI |
| Requested complete measurement | Eligible rows fit within the bound | Heuristic flag is used | State is `complete` with counts/rates | CLI/use case |
| Requested partial measurement | Eligible rows exceed the bound | Heuristic flag is used | State is `partial`, counts describe only scanned rows | CLI/SQLite |
| Requested failure | Exact succeeds and dedupe fails | JSON command runs | Exact metrics plus `failed` state are emitted; command returns error | CLI |
| Cleanup compatibility | Existing unbounded dry-run/apply/restore | Store dedupe commands run | Existing behavior remains unchanged | Existing regression suite |
| Apply safety | Bounded apply parameters | Use case runs | Request is rejected before datasource mutation | Use case |

### TDD plan
| Behavior | Red | Green | Refactor target |
|---|---|---|---|
| Four report states | Add CLI tests that fail against unconditional dedupe | Add typed input/state envelope and conditional measurement | Separate exact and heuristic envelope builders |
| Bounded datasource | Add integration assertions for total versus scanned | Count eligible rows and add deterministic LIMIT | Share agent predicates between count/load queries |

### Risks / rollback
- Procedural-logic risk: keep state construction in small report helpers; do not put output policy into the datasource.
- Premature-abstraction risk: add one numeric bound to the existing dedupe request instead of a new heuristic service.
- Migration / compatibility: no schema change. Existing callers use zero-value `MaxScanRows` and retain full-scan behavior.
- Rollback trigger: if bounded selection makes the maintenance command partial or mutable, revert immediately; only the report may pass a non-zero bound.

### Design checkpoint
- Risk is Medium: the public CLI JSON shape changes during an unreleased v0.30 feature cycle, and a shared maintenance request gains a bounded dry-run option.
- Default behavior becomes cheaper and safer; explicit heuristic output remains observable and failures do not erase exact metrics.
- The implementation is one PR because the type, adapter, CLI state, and tests form one atomic compatibility contract without a migration.
