# Issue #1504: bound and prioritize durable hook spool replay

## Structure-Behavior Design Note

### Requirement summary

- **Goal:** keep current hook delivery durable and low-latency while a large
  durable spool is present, and guarantee bounded, fair forward progress.
- **Current state:** every hook invocation decodes the complete spool and
  replays the oldest five before persisting the current delivery. A large or
  poisoned prefix can consume the host timeout and starve both new and later
  pending deliveries.
- **Expected behavior:** persist and attempt the current delivery first; remove
  its spool file immediately after a successful commit; only then select and
  decode a bounded replay batch. A failed replay remains durable but moves
  behind unattempted records.
- **Non-goals:** change the v1 spool payload, discard failed records, expose
  payload bodies, introduce a background daemon, or promise complete draining
  in one hook/doctor invocation.

### Conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Current delivery | unpersisted, durable, committed, failed/cancelled | persists before database work and commits before backlog work | success removes its own spool even if replay later fails or context expires |
| Replay queue | normal records followed by retried records | selects only a bounded filename-ordered batch | selection does not decode records outside the batch |
| Retry record | retained v1 payload with retry-ordered filename | atomically moves behind unattempted records after failure | payload and schema remain unchanged; failed rename never deletes the original |
| Drain result | selected, replayed, failed, remaining | reports bounded progress | unresolved records remain counted and durable |

### Responsibility assignment

| Responsibility | Owner | Reason to change | Not owner / reason |
|---|---|---|---|
| Persist/commit current delivery before replay | `runHookDurably` orchestration | hook lifecycle ordering changes | replay queue does not execute the current callback |
| Bounded selection and retry ordering | spool queue helpers | filesystem queue policy changes | individual hook adapters do not inspect directory layout |
| Decode and dispatch selected v1 records | spool drain + existing replay adapters | replay contract or supported command changes | doctor does not decode private payloads for progress counting |
| Exact fix progress | doctor spool fix | operator reporting changes | hook runtime emits only debug counts |

### Boundaries / interfaces

| Boundary or interface | Consumer | Hidden detail | Error contract |
|---|---|---|---|
| bounded replay path selection | drain | directory enumeration and retry filenames | returns at most the requested limit; directory failure prevents drain without deleting records |
| v1 record loader | drain | JSON payload bytes | malformed/unreadable record counts as failed and is retained/requeued |
| retry requeue | drain | atomic same-directory rename and random suffix | rename failure retains the original path and counts as failure |
| drain result | doctor/runtime | individual private payloads | exposes counts only |

### Behavior tests

| Behavior | Given | When | Then | Level |
|---|---|---|---|---|
| Current-first durability | large/failing backlog and a new delivery | durable hook runs | current is persisted, committed, and removed before replay |
| Cancellation after commit | current callback succeeds then cancels context | durable hook continues | current spool is removed and backlog remains untouched |
| Current failure | callback fails or context is cancelled before commit | durable hook exits fail-soft | current spool remains |
| Bounded decode | 10,000 pending v1 files | drain limit is five | at most five payload files are read/decoded |
| Fair retry | oldest selected record always fails and a later record is valid | two bounded drains run | failed record is retained but later valid record replays |
| v1 compatibility | existing schema version 1 record | selected for replay | existing adapter replays it unchanged |
| Doctor progress | mixed success/failure backlog | `doctor --fix` runs | message reports replayed, failed, and exact remaining counts |
| Privacy | payload contains private fields | progress/error is emitted | no payload body is printed |

### TDD plan

| Behavior | Red | Green | Refactor target |
|---|---|---|---|
| Current-first order | backlog callback observes current spool absent/committed too late | reorder persistence, current run, removal, then drain | keep fail-soft wrapper outside lifecycle ordering |
| Bounded selection | counting file reader observes reads above limit | select paths before loading records | isolate filesystem queue policy from replay dispatch |
| Fair retry | poison record is selected repeatedly | atomically rename failures into retry tail | keep v1 body immutable |
| Doctor counts | estimate is wrong with failures/unreadable records | count filename entries before/after bounded drain | share path filter with runtime selection |

### Risks / rollback

- **Procedural risk:** keep queue ordering and filesystem I/O out of individual
  hook adapters; `runHookDurably` remains orchestration only.
- **Premature abstraction:** introduce no general scheduler or background
  worker. Queue helpers are private to the existing hook spool.
- **Migration / compatibility:** no schema migration. Existing v1 JSON remains
  readable; retry is represented only by an atomic filename change.
- **Privacy:** selection and progress use filenames/counts. Payload bytes are
  read only for the bounded records being replayed and never rendered.
- **Rollback trigger:** current-delivery loss, deletion after failed commit,
  unbounded payload reads, retry starvation, or v1 incompatibility.
- **Rollback:** restore pre-replay ordering code only after disabling automatic
  backlog replay; durable files require no data migration.
- **PR plan:** one issue/one PR. Commit design/tests and runtime queue behavior
  separately from documentation if operator guidance changes.

### Design checkpoint

Proceed with a private filename-ordered spool queue. Normal timestamp filenames
remain the primary queue. Failed selected records are atomically renamed to a
`zz-retry-...` tail while preserving their v1 body, so subsequent bounded
drains advance through unattempted records. Persist, attempt, and clear the
current delivery before any opportunistic replay. Count paths without decoding
payloads for doctor progress.
