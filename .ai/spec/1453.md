# Issue #1453: persist run lineage and packet identity

## Structure-Behavior Design Note

### Requirement summary

- Purpose: make usage observations attributable to an opaque run hierarchy and work context without persisting prompts, responses, tool payloads, or fabricated identifiers.
- Current state: `Session` has parent/spawn lineage and `UsageObservation` has authoritative observation/session/source identity, but there is no run identity, parent-run relation, batch/ticket/PR/head attribution, or packet/tool-output size fact.
- Expected behavior: an adapter records one immutable run-lineage fact, may attach its namespaced run identity to a call/run observation, and can replay the same fact without mutation. Missing values remain absent. Conflicting identity, missing parent, self-parent, cyclic bundle input, and incomplete portability closure fail closed.
- Non-goals: host parsing, aggregate queries, provider request/retry semantics, packet body storage, tool name/argument/result storage, network telemetry, retroactive inference, or a session lifecycle rewrite.

### Adopted conceptual model

| Concept | State | Behavior | Constraint / invariant |
|---|---|---|---|
| Run identity | `(host, opaque run ID)` | identifies one execution within a host namespace | host is normalized through `UsageSource`; opaque ID is preserved byte-for-byte, non-empty/non-whitespace, and at most 512 UTF-8 bytes |
| Parent run | absent or `(parent host, parent run ID)` | forms a cross-host run tree | pair is all-or-none; known parent must already exist; no self-parent; append-only immediate-FK insertion preserves acyclicity by induction |
| Session attribution | absent or opaque session ID | correlates a run with a host session | absence is unknown; it is a correlation value, not an FK to Traceary `sessions`; no synthetic session is created |
| Work attribution | optional batch ID, ticket ref, repository, PR number, head SHA | supports later grouping | trimmed but not case-folded; empty values rejected; PR/head require repository; PR is positive; SHA is all lower-case hex and exactly 40 or 64 bytes |
| Packet identity | absent or `(sha256, bytes)` | compares the exact sanitized packet without storing it | pair is all-or-none; SHA is exactly 64 lower-case hex bytes; byte count is nullable non-negative `int64`, preserving known zero |
| Tool-output fact | absent or byte count | records total sanitized tool-output bytes admitted to the run context | nullable non-negative `int64`; known zero differs from unknown; no tool name, argument, result, or body exists |
| Usage-to-run link | absent or `(host, run ID)` in `UsageObservationDescriptor` | attributes call/run evidence to a durable run | stored in an immutable one-to-one association table; referenced run must exist; run host equals observation source host |
| Run-lineage lifecycle | immutable recorded fact | exact replay is a no-op | every semantic field participates in equality; any difference for the same namespaced identity is a typed conflict |

### Exact normalization and byte contracts

- Opaque run IDs are never trimmed, case-folded, prefixed, or concatenated. Validation rejects `""` and values whose `strings.TrimSpace` result is empty, but stores the original bytes unchanged.
- Every new textual lineage value must satisfy `utf8.ValidString` before entering the domain. SQLite restoration re-runs the same constructors, and NDJSON decode rejects replacement or invalid input rather than allowing JSON's invalid-byte replacement to change identity. Invalid UTF-8 fixtures cover domain construction, direct-SQL restore, and bundle decode; only valid UTF-8 is promised byte-for-byte preservation.
- Maximum lengths are UTF-8 bytes in Go (`len([]byte(value))`) and SQLite (`length(CAST(value AS BLOB))`), not rune/character counts.
- `batch_id`, `ticket_ref`, and `repository` are trimmed at construction and capped at 512, 512, and 2048 bytes respectively. Empty becomes unknown only when the optional value is absent; an explicitly present empty value is rejected.
- SHA values must satisfy length plus an all-character check equivalent to `value NOT GLOB '*[^0-9a-f]*'`.
- `packet_sha256` hashes the exact sanitized workspace packet bytes handed to the engine, before transport framing, compression, encryption, or provider encoding.
- `tool_output_bytes` is the sum of the sanitized tool-output payload bytes admitted to that run's engine context. Raw output that is dropped or redacted is not counted. If this boundary cannot be proved, the value is unknown.
- `packet_bytes=0` and `tool_output_bytes=0` use present optional integers in domain/NDJSON and non-NULL zero in SQLite; `omitempty` is never used on a non-pointer integer.

### Observation/run/session consistency

- A usage observation may link a run only for `call` or `run` scope. `session_snapshot` cannot carry a run identity because a cumulative session snapshot is not attributable to one run.
- The linked run host must equal `UsageObservationDescriptor.Source().Host()`.
- If the run has a known `session_id`, it must equal the observation's required `session_id`.
- A run with unknown session may still be linked to an observation. The observation's own session correlation remains authoritative for usage; the missing run session is not backfilled or inferred.
- `run_lineages.session_id` intentionally has no FK to `sessions`: one-shot and external host runs may have a valid opaque session correlation without a Traceary `sessions` row. Therefore run import does not require or imply session closure.
- Existing constructors remain source-compatible and create unknown run attribution. New explicit `NewUsageObservationDescriptorWithRunIdentity` and run-aware snapshot rejection tests are added rather than changing existing signatures.

### Persistence shape

Migration 28 adds two tables and does not alter migration-27 columns:

1. `run_lineages`
   - composite primary key `(host, run_id)`
   - nullable pair `parent_host`, `parent_run_id` with a composite immediate self-FK
   - nullable `session_id`, `batch_id`, `ticket_ref`, `repository`, `pull_request_number`, `head_sha`, `packet_sha256`, `packet_bytes`, `tool_output_bytes`
   - CHECK constraints for byte limits, optional pairs, strict hex, non-negative byte facts, PR/head repository dependency, and self-parent rejection
   - descriptor update/delete triggers reject all mutation; there is no recursive cycle trigger because append-only insertion plus immediate parent existence/self-check makes non-self cycles unreachable
2. `usage_observation_runs`
   - `observation_id` primary key/FK to `usage_observations`
   - composite `(run_host, run_id)` FK to `run_lineages`
   - insert trigger enforces call/run scope, host equality, and session equality when the run session is known
   - update/delete triggers reject mutation

This association-table design avoids adding a mutable `run_id` column to `usage_observations`, so migration 27's immutable descriptor trigger does not need to be weakened or replaced. Usage restore LEFT JOINs the association; no row means unknown run attribution.

No `recorded_at` field is introduced. It is neither required by #1453 nor part of host lineage semantics, and omitting it avoids ambiguous replay equality and false parent/child chronology constraints.

### Concurrency and replay

- `RunLineageRepository.Record` uses a serialized write transaction (`BEGIN IMMEDIATE`), restores any existing row, and asks `RunLineage.Reconcile` for exact replay vs typed conflict.
- It never treats `INSERT OR IGNORE` alone as idempotency evidence.
- Concurrent identical writers yield one applied and one already-applied result. Concurrent differing writers yield one applied and one typed conflict; the winning row remains unchanged.
- Conflict errors contain only namespaced run identity and a closed conflicting-field label, never repository, ticket, hash, or field values.

### Bundle/import contract

- Registry order is `sessions -> run_lineages -> usage_observations -> ...`; `sessions` remain independent, while runs must precede usage association insertion.
- Every newly exported manifest-v2 bundle includes `run_lineages` even when its row count is zero. Older readers reject the unknown table rather than silently losing lineage. New readers continue accepting old v2 bundles without this optional table.
- Unfiltered export includes every durable run lineage, including standalone runs not referenced by usage.
- Filtered export starts from the run identities referenced by selected usage observations and includes their complete ancestor closure only. It excludes unrelated children and siblings.
- Export validates uniqueness, parent existence, and acyclicity while calculating closure. Missing parent/cycle fails bundle creation.
- Encoding is deterministic parent-before-child. Import does not trust NDJSON order: it validates and topologically sorts the complete run table, failing missing parents/cycles before writes.
- Exact existing lineage is already-applied. Different same-ID content, missing parent, cycle, session/host mismatch, or incomplete association fails the entire transaction under every conflict policy. `skip`, `replace`, session backfill, and missing-parent tolerance never weaken lineage integrity.
- `BundleImportResult` adds `RunLineagesImported` and `RunLineagesSkipped`.
- Compatibility fixtures explicitly prove: a new reader accepts an old v2 bundle with no `run_lineages`; every new export includes the zero-row entry; a usage/run association without its table or referenced run rolls the whole import back; and an old-reader registry rejects the new unknown table rather than ignoring it.

### Concrete Go boundaries

- `types.RunIdentity` is an immutable value object with `Host() string` and `RunID() string`. `types.Optional[types.RunIdentity]` represents unknown attribution.
- `UsageObservationDescriptor.RunIdentity()` returns that optional. Existing `NewUsageObservationDescriptor` and `NewUsageSnapshotDescriptor` signatures remain unchanged and return unknown attribution.
- `NewUsageObservationDescriptorWithRunIdentity` is the only new run-aware constructor and accepts call/run scope; there is no run-aware snapshot constructor.
- `model.RunLineageRepository` exposes `Record(context.Context, *RunLineage)` and `FindByIdentity(context.Context, types.RunIdentity)`; a small `RunLineageUsecase` delegates to it.
- `BundleEventRepository` explicitly adds `ListBundleRunLineages`; `BundleImportTransaction` adds `ImportRunLineage`; `bundleExportInputRows` adds `RunLineages`; and the table registry adds `bundleRunLineagesTable` plus result counters.
- Existing bundle fakes/mocks are updated deliberately, and `go test ./...` is the compile-compatibility gate for all consumers. No type assertion or optional hidden repository capability is used.

### Risks / rollback

- Procedural-logic risk: bundle or datasource code could become the owner of lineage semantics. Mitigation: domain aggregate owns semantic equality; graph closure/topology is an explicit bundle boundary behavior; SQL enforces transactionally observable integrity.
- Primitive-obsession risk: work/head/packet fields could become unvalidated strings. Mitigation: typed constructors and mirrored byte/pair/hex constraints.
- Premature-abstraction risk: no provider-specific run hierarchy, arbitrary metadata JSON, generic key/value table, or body-bearing packet/tool field is introduced.
- Privacy: SHA-256 is an identifier, not anonymization. It is never logged in conflicts and no packet body is persisted.
- Migration/compatibility: migration 28 is additive; association absence means unknown. A v27 fixture must preserve usage, `ended_at`, `runtime_mode`, `terminal_reason`, parent/spawn metadata, and existing bundle behavior.
- Import ordering: ancestors restore before children, then usage. Missing ancestors/cycles fail rather than backfill invented lineage.
- Rollback: stop new lineage writes first. Schema downgrade is unsupported; reverting to an older binary after lineage writes requires restoring a pre-migration backup. The additive tables can otherwise remain unused.
- Rollback trigger: any cross-host collision, cycle, same-ID semantic drift, private-body persistence, broken v27 restore, severed filtered ancestor chain, policy-weakened conflict, or changed one-shot/interactive lifecycle blocks adapter rollout.

### PR scope and design checkpoint

- One PR closes only #1453.
- It includes domain types/model, migration 28, SQLite repository, optional usage association, encrypted bundle compatibility, import counters, and behavior tests.
- It does not parse host files and does not expose aggregate CLI/MCP output.

Design checkpoint decision: adopt composite `(host, opaque run ID)` identity, append-only immediate parent FKs, one immutable usage/run association table, nullable typed work/packet facts, and fail-closed ancestor closure. Do not add arbitrary metadata JSON, inferred IDs, body-bearing fields, timestamps without source semantics, or a mutable column on `usage_observations`.
