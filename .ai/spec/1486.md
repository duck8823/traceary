# Issue #1486: retention root apply-readiness evidence

## Requirements and risk

- High-risk operator-safety fix: `doctor` must warn when exact file-retention apply would reject the inspected root.
- Root ownership and group/other-write evidence must come from the same open root descriptor used for inventory.
- Inspection remains read-only and must not weaken or bypass apply validation.
- Unsupported or unknown evidence fails closed as a warning.

## Concept model

| Concept | State | Rule |
|---|---|---|
| RootAccessEvidence | apply state, caller ownership, group/other write bit | derived once from the open root descriptor |
| RootApplyState | `eligible`, `unsafe_owner`, `unsafe_permissions`, `unsupported` | exact apply is eligible only in the first state |
| CapacityStatus | inventory readiness plus root access evidence | recovery readiness and apply-root readiness remain distinct |

## Responsibilities and boundaries

| Responsibility | Owner |
|---|---|
| derive UID/mode evidence from `fstat` | Unix filesystem adapter |
| preserve the exact apply rejection | filesystem executor using the same evidence helper |
| carry evidence without filesystem parsing | application status DTO |
| render pass/warn and remediation | doctor presentation |

The CLI must not inspect path permissions itself. The adapter supplies factual descriptor-bound evidence; doctor only maps the application status to operator output.

## Observable behavior tests

1. Caller-owned root without group/other write access reports `eligible` and doctor may pass.
2. Group-writable root reports `unsafe_permissions`; inventory remains read-only and doctor warns.
3. Unsupported/unknown evidence reports `unsupported` and doctor warns.
4. Exact apply still rejects the same group-writable root before creating retention state.
5. Copied-store doctor output proves safe and unsafe roots are distinguished.
