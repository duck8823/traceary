# v0.30.0 requested/effective interval dogfood

[日本語](./v0.30.0-interval-semantics-dogfood.ja.md)

## Scope

Issue #1432 was exercised with a locally built binary and a disposable fresh
SQLite store. The live Traceary store was not modified.

## Commands

```sh
go build -o /private/tmp/traceary-issue-1432 .
/private/tmp/traceary-issue-1432 report \
  --db-path /private/tmp/traceary-issue-1432-qa.db \
  --workspace dogfood \
  --from 2026-03-08 --to 2026-03-08 \
  --timezone America/New_York --json
/private/tmp/traceary-issue-1432 list \
  --db-path /private/tmp/traceary-issue-1432-qa.db \
  --workspace dogfood \
  --from 2026-03-08 --to 2026-03-08 \
  --timezone America/New_York --json
/private/tmp/traceary-issue-1432 search absent \
  --db-path /private/tmp/traceary-issue-1432-qa.db \
  --workspace dogfood \
  --from 2026-03-08 --to 2026-03-08 \
  --timezone America/New_York --json
```

## Results

- The report preserved `requested_to=2026-03-08`.
- The effective interval was
  `[2026-03-08T05:00:00Z, 2026-03-09T04:00:00Z)`, a 23-hour calendar day
  across the New York spring DST transition.
- Text output displayed the requested calendar end on the primary `Period`
  line and labeled the effective exclusive boundary separately.
- `list` and `search` accepted the same date-only/timezone inputs on the fresh
  store and returned empty JSON arrays without changing the interval rules.
- `--timezone Mars/Olympus` and the host-dependent special name `Local` were
  rejected before any query with exit code 1 and an explicit invalid-timezone
  error.
- Unit coverage also pins the 25-hour fall DST day, RFC3339 exact-exclusive
  behavior, omitted-upper-bound snapshots, and matching CLI/MCP criteria.
