## Follow-up finding と対応

1. #1498 / PR #1503 は、durable delivery が queue に残っている Codex completion evidence を「存在しない」と誤報する問題を修正しました。
2. #1504 / PR #1505 は、上限なし/retry-first の spool replay を修正しました。current delivery を先に処理し、retry work を上限付きかつ公平にし、並行 consumer の ownership を分離し、stale claim を安全に復旧します。
3. #1506 / PR #1507 は、Grok headless observation の時刻が Unix epoch になる問題を修正しました。マージ済み candidate は UTC の ingest time を割り当て、portable replay の同値判定からローカル observation/finalization time だけを除外します。

3 件とも、決定的検証と本文なし live dogfood 後にマージしました。PR #1503、#1505、#1507 では必要な独立 reviewer topology を起動できませんでしたが、maintainer が各マージで残リスクを明示的に承認しています。

## 運用上の観測

最終 `doctor --fix --dry-run` では、読み取り可能な未処理 spool record 8,678 件、読み取り不能 0 件、SQLite store 3.2 GiB を確認しました。現在インストール済みの v0.31 binary は Traceary workspace を `hook_spool_backlog`、committed usage observation なしと報告します。これは upgrade 前の運用負債であり、未修正の v0.32 code path ではありません。v0.32 candidate は別途、current event を上限付き retry batch より先に処理し、backlog を減らすことを確認済みです。`doctor --fix` 全体は無関係な session/config maintenance も行うため、意図的に適用していません。

release 後には、installed v0.32 binary が backlog を drain し、新しい Codex completion が backlog を増やさないことを確認します。store compaction は operator action（`traceary store gc --dry-run`）のままであり、usage accounting の意味を変える release 条件ではありません。

## Release 判定

合格。QA で見つかった release-blocking finding はすべて Issue 化、実装、マージ、再検証済みです。修正後の usage-capable live probe は availability 100% で、unavailable / partial path も明示分類されています。重複加算と private body 依存はありません。過去の spool backlog は upgrade 後に監視する運用リスクであり、release を止める correctness defect ではありません。
