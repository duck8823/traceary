## 決定的なプロセス終了結果

ラッパーからセッション ID を明示して 4 個のシェルプロセスを実行した。

```sh
traceary session run --db-path /private/tmp/traceary-1436.db --session-id dogfood-success -- sh -c 'printf success-output'
traceary session run --db-path /private/tmp/traceary-1436.db --session-id dogfood-failure -- sh -c 'exit 7'
traceary session run --db-path /private/tmp/traceary-1436.db --session-id dogfood-timeout --timeout 20ms -- sh -c 'sleep 1'
traceary session run --db-path /private/tmp/traceary-1436.db --session-id dogfood-signal -- sh -c 'kill -TERM $$'
```

ラッパーの終了コードは順に `0`、`7`、`124`、`143` だった。SQLite を確認し、全行が `runtime_mode=one_shot`、終了理由が `success`、`failure`、`timeout`、`signal`、各セッションの `session_ended` が 1 件、`PRAGMA integrity_check=ok` であることを確認した。出力先の書き込み失敗はプロセスアダプターのテストで再現し、`aborted_stream` と終了コード `74` を確認した。

## Codex exec 統合

ビルドした Traceary を `PATH` の先頭に置き、空の一時 workspace から read-only、ephemeral の Codex を実行した。

```sh
NO_BROWSER=1 traceary session run \
  --db-path /private/tmp/traceary-codex-dogfood.db \
  --session-id dogfood-codex-exec \
  --agent codex \
  --workspace /private/tmp/traceary-codex-dogfood \
  -- codex exec --ephemeral --skip-git-repo-check --sandbox read-only \
  -C /private/tmp/traceary-codex-dogfood \
  -o /private/tmp/traceary-codex-last.txt \
  'Reply exactly PASS'
```

- プロンプト SHA-256: `3ea9dc67727f144161bbf12b01968e0db4aee65ea7ae4e6d9d66fa17c4860601`
- 出力: `PASS`
- セッション行: `dogfood-codex-exec` 1 件、`one_shot`、`success`
- すべての hook event がラッパーのセッション ID を使用し、ホスト側 ID の別セッションは作られなかった。
- Codex `Stop` は transcript 境界を作成し、唯一の `session_ended` はラッパーが記録した。
- ストア整合性: `ok`

最初の sandbox 内実行は、ローカルの in-process app-server を初期化できずモデル実行前に失敗した。認証済みホスト通信で 1 回だけ再試行し成功した。リポジトリのソース、認証情報、トークン値、ブラウザ状態、実運用データは送信していない。

## 判定

v0.30.0 の次工程へ進める。完了判定は監督対象プロセスの終了だけに基づき、対話型 `Stop` の挙動を維持し、Codex hook をラッパー ID に対応付け、終了遷移を 1 回だけ記録できている。
