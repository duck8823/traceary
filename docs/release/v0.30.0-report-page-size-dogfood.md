# v0.30.0 report page-size dogfood

[日本語](./v0.30.0-report-page-size-dogfood.ja.md)

## Finding

The release-wide v0.30.0 QA pass ran `report` against an empty disposable store with `--page-size 9223372036854775807`. The pre-fix binary exited 2 with `runtime error: makeslice: cap out of range` from `loadCappedReportRows`. Follow-up Issue #1479 tracks the release-blocking availability defect.

## Fixed-binary verification

The same command was repeated with the Issue #1479 binary:

```sh
traceary report \
  --db-path /private/tmp/traceary-v030-qa.db \
  --from 2026-07-01 \
  --to 2026-07-01 \
  --page-size 9223372036854775807 \
  --json
```

The fixed binary exited 1 with the normal validation error `page size must be less than or equal to 100000`. It emitted no panic or runtime stack. Unit, CLI, and MCP tests also cover the maximum accepted value, maximum plus one, and a huge integer.

## Verdict

Pass. CLI and MCP share the 100,000-row internal page maximum. Invalid resource controls fail before report query execution, while normal full/partial aggregation behavior is unchanged.
