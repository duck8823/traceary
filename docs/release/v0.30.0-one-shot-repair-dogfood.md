# v0.30.0 one-shot repair dogfood

## Scope and safety boundary

- Issue: #1434
- Binary: `/private/tmp/traceary-1434`
- Copied store: `/private/tmp/traceary-1434-backlog.db`
- Evidence manifest: `/private/tmp/traceary-1434-backlog-evidence.json`
- Manifest SHA-256: `67803a7db694cf83b34aac1c71f3319592327119988b1f7b5be13ed76937a343`
- Candidate entries: 10,170
- No production store was modified.
- Raw session IDs and the evidence manifest remain outside the repository.

The current store already contained the 10,170 observed backlog sessions as
`legacy_unknown` terminal rows because stale-session garbage collection had run
after the original batch. A consistent SQLite backup was copied to
`/private/tmp`, the existing `ended_at` values were retained as conservative
operator-attested process completion times in the manifest, and only the copied
candidate rows were reconstructed to their observed pre-GC active state. This
created a faithful, reversible repair rehearsal without mutating live data or
inferring one-shot mode from transcript text, idle time, or workspace membership.

## Dry-run

The dry-run reported all 10,170 explicit manifest entries as `eligible` and
changed zero rows. The copied target set remained 10,170 active interactive
sessions with no `one_shot_repair` events. The database SHA-256 was identical
before and after the command (`72f5222376c911c0072c4f59c24a3a6c2bcc5c6063c685584a2bfada598e88a0`).

## Apply and idempotency

Apply required and created
`/private/tmp/traceary-1434-backlog-before-apply-final.db` before starting the atomic
repair transaction. It applied all 10,170 candidates:

| Metric | Before | After |
|---|---:|---:|
| Active | 10,335 | 165 |
| Stale | 10,181 | 11 |
| Completed success | 0 | 10,170 |
| Typed failure | 0 | 0 |

SQLite inspection confirmed 10,170 target rows with
`runtime_mode=one_shot` and `terminal_reason=success`, plus exactly 10,170
distinct `session_ended` events sourced from `one_shot_repair`. The aggregate
state of all 11,082 non-target sessions was unchanged, and
`PRAGMA integrity_check` returned `ok`.

A second apply created another required pre-apply backup, classified all 10,170
entries as `already_terminal`, changed zero rows, and left the repair-event
count at exactly 10,170. Store integrity remained `ok`.

## Rollback rehearsal

The required pre-apply backup was restored to a separate database path. The
restored target set contained 10,170 active interactive sessions, zero repair
events, the same non-target aggregate state, and
`PRAGMA integrity_check=ok`.

## Decision

The evidence-backed repair is acceptable for the v0.30.0 sequence. It is a
no-write dry-run by default, requires a successful backup before apply, explains
every supplied candidate, applies all eligible transitions in one transaction,
does not touch sessions omitted from the manifest, is idempotent, and has a
verified restore path.
