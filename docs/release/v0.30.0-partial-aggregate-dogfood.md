# v0.30.0 partial aggregate dogfood

[日本語](./v0.30.0-partial-aggregate-dogfood.ja.md)

## Scope

Issue #1438 aligns CLI and MCP report aggregates with explicit complete/partial metadata. This dogfood used the locally built binary and a disposable SQLite database containing three sessions and three command audits.

## Procedure

1. Build the current worktree binary.
2. Record three sessions and one successful `go test ./...` audit in each session.
3. Run `traceary report --json --page-size 1` over a fixed interval.
4. Run the same report with `--result-cap 1`.
5. Assert the complete and partial provenance with `jq`.
6. Run the MCP `get_report` handler and registry integration tests.

## Observations

| Scenario | Result |
|---|---|
| Page size 1, no result cap | `aggregation.coverage=complete`; source counts were sessions 3, events 6, commands 3; numeric ratios remained present |
| Page size 1, result cap 1 | all three sources reported `partial`, `response_truncated=true`, and `truncation_reason=result_cap` with an observed timestamp range |
| Partial percentages | `prompt_transcript_missing_ratio` and `failure_rate` were absent rather than calculated from the prefix |
| Stored bodies | the SQLite integration test stored 256 KiB command outputs while the report query used body-free projections |
| MCP surface | `get_report` returned the shared `ReportSnapshot` schema and the nine-tool registry golden matched |

## Verdict

Pass. Internal page size no longer changes aggregate completeness. A caller can request sampling only through the explicit result cap, and the response makes that loss of coverage machine-readable.
