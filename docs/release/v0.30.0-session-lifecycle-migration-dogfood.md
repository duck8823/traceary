# v0.30.0 session lifecycle migration dogfood

[日本語](./v0.30.0-session-lifecycle-migration-dogfood.ja.md)

Date: 2026-07-22

Issue: #1430

## Method

An APFS copy-on-write clone of the existing 3.06 GiB copied local store from the v0.30 workspace-report QA was used. The Issue #1430 binary applied migration 24 only to that disposable clone. The live store and the earlier QA source copy were not changed.

Before and after checks used aggregate counts and `PRAGMA integrity_check`. No event body, session identifier, credential, or raw row is recorded here or shared with an external reviewer. A second small disposable store tested binary rollback compatibility: the current binary created schema 24, installed v0.29.0 started and ended a session without lifecycle columns, and the current binary then restored that ended aggregate and updated its label successfully.

## Copied-store population

- database size: 3.06 GiB
- events: 212,023
- sessions: 21,228
- active sessions: 161
- ended sessions: 21,067
- starting schema: 23 migrations

## Migration measurement

Command: `/usr/bin/time -l traceary-under-test store init --db-path <disposable-copy>`

- elapsed: 19.51 seconds for the final trigger-enabled migration (19.63 seconds in the preceding full measurement)
- maximum resident set size: 36,782,080 bytes (35.1 MiB)
- resulting schema: 24 migrations
- runtime modes: 21,228 `interactive`
- terminal reasons: 161 empty active values; 21,067 `legacy_unknown` ended values
- invalid active-with-reason rows: 0
- invalid ended-without-reason rows: 0
- event count after migration: 212,023
- `PRAGMA integrity_check`: `ok`
- active-session terminal guards: both insert and update triggers present; a direct active-with-reason insert was rejected

The timing is a one-time migration measurement on this local APFS clone, not a general performance guarantee. The migration is additive and updates only the two lifecycle columns; all earlier aggregate fields were covered by automated preservation tests.

## Rollback compatibility

Installed `traceary 0.29.0` successfully started and ended a session in the final schema-24 disposable database because both new columns have safe defaults and the guards allow an ended row with an empty reason. The guards reject only a terminal reason attached to an active row. That old end left `terminal_reason` empty, as expected. The Issue #1430 binary restored it conservatively as `legacy_unknown` and completed a label update without changing the end marker. The disposable database remained integral.

## Release decision

The lifecycle migration is acceptable for the v0.30.0 implementation sequence. Historical sessions are never inferred as one-shot, historical outcomes are not fabricated, the source data and row counts are preserved, and rolling back to the installed v0.29.0 binary does not make the additive schema unwritable.
