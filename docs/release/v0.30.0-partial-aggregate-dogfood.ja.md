# v0.30.0 部分集計 dogfooding

[English](./v0.30.0-partial-aggregate-dogfood.md)

## 対象

Issue #1438 では、CLI と MCP のレポート集計に完全・部分を明示する情報を追加します。この dogfooding では、現在の worktree からビルドしたバイナリと、3 session・3 command audit を入れた使い捨て SQLite DB を使いました。

## 手順

1. 現在の worktree からバイナリをビルドする。
2. 3 session を作り、それぞれに成功した `go test ./...` audit を 1 件記録する。
3. 固定期間に対して `traceary report --json --page-size 1` を実行する。
4. 同じ条件に `--result-cap 1` を追加して実行する。
5. `jq` で完全・部分の来歴を検証する。
6. MCP `get_report` handler と tool registry の統合テストを実行する。

## 観測結果

| 条件 | 結果 |
|---|---|
| ページサイズ 1、結果上限なし | `aggregation.coverage=complete`。データ源の件数は session 3、event 6、command 3 で、数値の割合も維持した |
| ページサイズ 1、結果上限 1 | 3 データ源すべてが `partial`、`response_truncated=true`、`truncation_reason=result_cap` と観測時刻範囲を返した |
| 部分集計時の割合 | 先頭部分から計算せず、`prompt_transcript_missing_ratio` と `failure_rate` を省略した |
| 保存本文 | SQLite 統合テストでは 256 KiB の command output を保存したまま、本文を含まない projection でレポートを取得した |
| MCP | `get_report` が共通の `ReportSnapshot` スキーマを返し、9 tool の registry golden と一致した |

## 判定

合格。DB 内部のページサイズは集計の完全性を変えません。部分取得は明示的な結果上限でだけ要求でき、応答から網羅性の喪失を機械的に判定できます。
